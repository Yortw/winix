# ADR: Winix Man Page Viewer

**Date:** 2026-03-31
**Status:** Proposed
**Related design:** `2026-03-31-man-design.md`

---

## 1. Author in Markdown, ship groff, render groff on all platforms

**Context:** Man pages need to be authored for winix tools. Raw groff is unpleasant to write. The tool also needs to render pages on platforms (Windows) where groff isn't installed.

**Decision:** Author man pages in Markdown. Convert to groff at build time using `pandoc`. Ship only the generated groff files. The winix `man` command renders groff natively on every platform — no conditional rendering paths.

**Rationale:** Single source format (Markdown) for authoring comfort. Single distribution format (groff) for consistency. Single rendering engine across all platforms. On Linux/macOS the shipped `.1` files also work with system `man`, so winix `man` is a convenience, not a requirement.

**Trade-offs Accepted:** Build-time dependency on `pandoc` (or equivalent). Mitigated by committing generated files so builds don't require it — only regeneration does. The Markdown source must follow `pandoc`'s man page conventions (definition lists for options, specific heading structure).

**Options Considered:**
- *Ship Markdown, render Markdown on Windows, groff on Unix:* rejected — two rendering paths means two sets of bugs and inconsistent output.
- *Write groff directly:* rejected — hostile authoring experience with no practical benefit over Markdown + conversion.
- *Skip groff entirely, render Markdown everywhere:* rejected — breaks compatibility with system `man` on Linux/macOS. Third-party man pages (Git, coreutils) are all groff.

## 2. Build a groff renderer, not a full groff interpreter

**Context:** Groff is a full typesetting language. Man pages use a small subset via the `man` and `mdoc` macro packages.

**Decision:** Implement a renderer that handles the `man` macro set (MVP) and `mdoc` macro set (phase 2). Do not implement the full groff language (conditionals, string registers, diversions, traps, arithmetic).

**Rationale:** The `man` macro set covers ~95% of real-world man pages. Adding `mdoc` reaches ~99%. Full groff would require building a typesetting engine for the remaining 1% of obscure pages. The effort-to-value ratio is extremely poor.

**Trade-offs Accepted:** A small number of man pages using raw groff requests or exotic macros won't render correctly. These are rare in practice — most man page authors use standard macros. Pages that fail can be viewed with system `man` on Linux/macOS.

**Options Considered:**
- *Wrap an existing groff binary:* rejected — the whole point is not needing groff installed, especially on Windows.
- *Use a groff library (e.g. mandoc's C library):* rejected — C interop adds complexity and platform-specific build concerns. The macro subset is small enough that a native C# implementation is cleaner and more maintainable.

## 3. Three-stage rendering pipeline (lexer → macro expander → terminal renderer)

**Context:** Need to parse groff source and produce formatted terminal output. Two different macro sets (`man` and `mdoc`) must produce the same kind of output.

**Decision:** Three-stage pipeline: lexer (tokenises raw groff), macro expander (interprets macros into an intermediate representation of styled document blocks), terminal renderer (emits ANSI text from IR). Two macro expander implementations share the same IR and renderer.

**Rationale:** The IR decouples parsing from rendering. Adding `mdoc` support means writing a new expander, not a new renderer. The IR also enables `--json` output (serialise blocks instead of ANSI text) and future output formats (HTML, PDF) without touching the parser.

**Trade-offs Accepted:** Slightly more complex than a single-pass renderer. Worth it for the clean separation and reuse.

**Options Considered:**
- *Single-pass parser+renderer:* rejected — would need to be duplicated for `mdoc`, and adding new output formats would require rewriting.
- *AST instead of IR:* considered — an AST preserves macro structure, but the renderer doesn't need macro semantics, just styled blocks. The IR is a simpler, more focused abstraction.

## 4. Layered page discovery with auto-detection of well-known locations

**Context:** Man pages can come from many sources — winix tools, Git for Windows, MSYS2, Homebrew, system packages. The viewer needs to find them without manual configuration.

**Decision:** Three-layer search: bundled pages (alongside binary), `MANPATH` entries, auto-detected well-known locations. First match wins. Auto-detection covers Git for Windows, MSYS2, Homebrew, and standard Unix paths.

**Rationale:** Zero-config for winix tools (bundled). Power users can extend via `MANPATH`. Auto-detection means `man git-log` works on Windows if Git for Windows is installed, without the user configuring anything.

**Trade-offs Accepted:** Auto-detection heuristics (registry checks, path probing) may find stale or unexpected installations. Mitigated by the `--manpath` flag for debugging the effective search path.

**Options Considered:**
- *`MANPATH` only:* rejected — requires manual setup on Windows where there's no convention.
- *Bundled only (winix pages):* rejected — too limited for the general-purpose viewer goal.

## 5. Pager chain with built-in minimal fallback

**Context:** Man pages need a pager for comfortable reading. Windows doesn't reliably have `less` installed.

**Decision:** Pager priority: `$MANPAGER` → `$PAGER` → sibling winix `less` → system `less` → built-in minimal pager → raw stdout (if piped). The built-in pager supports scrolling, page up/down, home/end, `/` search, and `q` quit.

**Rationale:** Respects user preference (env vars). Automatically uses winix `less` if/when it ships. Falls back gracefully. The built-in pager is minimal but sufficient — reading a man page shouldn't require installing a separate tool.

**Trade-offs Accepted:** The built-in pager is basic compared to `less`. This is intentional — it's a fallback, not a replacement. If winix `less` is built later, it supersedes the built-in automatically via the priority chain.

**Options Considered:**
- *No built-in pager, dump to stdout:* rejected — unusable for long pages on a system with no `less`.
- *Full `less` clone built-in:* rejected — massive scope for a fallback. Better to build `less` as its own tool if warranted.
- *Depend on winix `less` as hard requirement:* rejected — couples two tools unnecessarily and doesn't help until `less` is built.

## 6. ANSI colour and OSC 8 hyperlinks, capability-detected

**Context:** Winix tools use colour and hyperlinks as standard terminal features. Man page output should match this quality.

**Decision:** Section headings get colour. Bold/italic use ANSI attributes. Cross-references (e.g. `grep(1)`) get OSC 8 hyperlinks in supporting terminals. All capability-detected via ShellKit's `ConsoleEnv` — no colour when piped, no OSC 8 when unsupported, `NO_COLOR` respected.

**Rationale:** Consistent with the rest of the winix suite. Colour aids scanning. Hyperlinked cross-references are a genuine UX improvement — clicking `see grep(1)` to jump there is how man pages should have always worked.

**Trade-offs Accepted:** OSC 8 URI scheme for man pages (`man:grep(1)`) is non-standard. Terminals won't handle it without a custom handler. Practical impact is low — the link is there for terminals that support it, and degrades to plain styled text otherwise.

**Options Considered:**
- *Plain text only:* rejected — below winix's quality bar.
- *Colour without hyperlinks:* considered as MVP, but hyperlinks are cheap to add once colour rendering works, and the value is high.

## Decisions Explicitly Deferred

| Topic | Why deferred |
|-------|-------------|
| `mdoc` macro implementation details | Phase 2 — need MVP experience with `man` macros first to validate the IR design |
| `man -k` / `apropos` indexing format | Phase 3 — index design depends on the scale of pages being searched |
| OSC 8 URI scheme for cross-references | Needs experimentation — may use `man:name(section)` or `x-man-page:` (macOS convention) or just re-invoke `winix man` |
| Winix `less` as a separate tool | Separate brainstorm — `man` doesn't depend on it |
| Man page section for winix tools | Likely section 1 for all, but should be confirmed during authoring |
| Colour theme configurability | Start with sensible defaults, add config if users request it |
