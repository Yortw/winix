# ADR: retry Tool Design Decisions

**Date:** 2026-04-16
**Status:** Proposed
**Context:** Design for `retry`, a command retry tool for the Winix CLI suite.
**Related:** [retry design doc](2026-04-16-retry-design.md)

---

## 1. Delay Input Format

**Context:** How should users specify the delay between retries?

**Decision:** Duration strings via the existing `DurationParser` from `Winix.FileWalk` (e.g. `2s`, `500ms`, `1m30s`).

**Rationale:** Already built and tested. Consistent with how `peep` accepts intervals. More expressive than bare seconds for sub-second and multi-unit delays.

**Trade-offs Accepted:** Slightly more to type than bare numbers for simple cases (`2s` vs `2`).

**Options Considered:**
- Plain seconds (simpler but less expressive, inconsistent with peep)
- Both formats with auto-detection (ambiguity risk, more parsing code)

---

## 2. Default Backoff Strategy

**Context:** What backoff strategy should apply when `--backoff` is omitted?

**Decision:** `fixed` — each retry waits the same delay.

**Rationale:** retry is a local-first tool for flaky commands. Fixed delay is simple, predictable, and what most users expect. Users hitting APIs can explicitly opt into `--backoff exp`.

**Trade-offs Accepted:** Network-heavy use cases require an extra flag.

**Options Considered:**
- Exponential default — better for APIs but surprising for local commands

---

## 3. Dual Exit-Code Filtering: --on and --until

**Context:** Users need to control which exit codes trigger retries vs which stop the loop.

**Decision:** Two complementary flags:
- `--on X,Y` — whitelist: retry only on these codes
- `--until X,Y` — target: stop when you see one of these codes
- Both together = usage error
- Default (no flags) = implicit `--until 0`

**Rationale:** These are inverse views of the same problem. `--on` is natural for "these specific failures are transient." `--until` is natural for "keep polling until ready." Having both avoids forcing users into the wrong mental model.

**Trade-offs Accepted:** Two related flags add learning surface. Mitigated by the default covering the common case (retry until success) without any flags.

**Options Considered:**
- `--on` only — forces users to enumerate "success" codes negatively
- `--fail-fast` (enumerate permanent failures) — less intuitive than whitelisting transient ones
- Single `--exit-code` with mode modifier — more complex, no real benefit

---

## 4. Exit Code 0 Not Special-Cased With --until

**Context:** When `--until 1,121` is specified and the command exits 0, should 0 implicitly stop the loop?

**Decision:** No. 0 is not special-cased. `--until 1,121` retries on exit 0 because 0 is not in the target set.

**Rationale:** This enables polling scenarios where exit 0 means "not ready" and a specific non-zero code means "ready." The rule is simple and consistent: stop on codes in the set, retry on everything else. Users who want to stop on 0 include it: `--until 0,1,121`. The default (no flags) already handles the overwhelmingly common "retry until success" case.

**Trade-offs Accepted:** Users writing `--until 121` might be surprised that exit 0 retries. Mitigated by clear documentation and the fact that `--until` is an advanced flag — the default just works.

**Options Considered:**
- Always treat 0 as implicit stop — breaks polling use cases, inconsistent rule

---

## 5. Jitter as Opt-In Flag

**Context:** Should exponential backoff include jitter by default?

**Decision:** Jitter is opt-in via `--jitter`. Works with any backoff strategy, not just exponential.

**Rationale:** retry is local-first. Deterministic delays are easier to reason about and test. Thundering herd is rare for local commands. Users doing API/network retries can add `--jitter`.

**Trade-offs Accepted:** Network use cases need an extra flag for best-practice behaviour.

**Options Considered:**
- Always jitter on exponential — right for distributed systems, surprising for local
- Jitter on by default with `--no-jitter` opt-out — retry is local-first, default should be predictable

---

## 6. Synchronous Sleep for Delays

**Context:** How should the retry loop wait between attempts?

**Decision:** `Thread.Sleep` (synchronous), not `Task.Delay` (async).

**Rationale:** The tool blocks on one child process at a time. There is no concurrent work to overlap during the delay. Synchronous sleep avoids async colouring the library API and keeps the implementation straightforward. CancellationToken support for Ctrl+C uses the overload that accepts a token.

**Trade-offs Accepted:** Cannot overlap delay with other work. Not needed — there is no other work.

**Options Considered:**
- Async throughout — unnecessary complexity for a sequential tool

---

## Decisions Explicitly Deferred

| Topic | Why Deferred |
|-------|-------------|
| `--max-delay` cap | No user demand yet. Easy to add later without breaking changes. |
| `--timeout` per-attempt | Adds process kill complexity. Revisit if requested. |
| `--forever` / infinite retries | `--times 0` could mean this, but needs careful thought about Ctrl+C and runaway loops. |
| Shared process-spawning code with timeit | Both use the same `ProcessStartInfo` pattern. Extract to ShellKit if a third tool needs it. |
