# ADR: clip Tool Design Decisions

**Date:** 2026-04-17
**Status:** Proposed
**Context:** Design for `clip`, a cross-platform clipboard bridge for the Winix CLI suite.
**Related:** [clip design doc](2026-04-17-clip-design.md)

---

## 1. Shell-Out on Linux and macOS; Native P/Invoke on Windows

**Context:** How should `clip` talk to the clipboard on each platform?

**Decision:** Use direct P/Invoke on Windows; shell out to platform helpers on Linux (`wl-copy`/`wl-paste`, `xclip`, `xsel`) and macOS (`pbcopy`/`pbpaste`).

**Rationale:** Windows has a stable user32 clipboard API that's AOT-friendly and essentially free to call. Linux has no native clipboard — it's a protocol layered on X11 and Wayland. Implementing the ICCCM selection dance and Wayland's `wl_data_device_manager` from scratch in C# would be weeks of work to reproduce what `xclip` and `wl-copy` already do reliably. macOS `pbcopy`/`pbpaste` are always present. Shelling out keeps the implementation small, the behaviour consistent with what users already expect on each platform, and leaves the door open for native backends later (deferred as v2 work).

**Trade-offs Accepted:** On Linux, the binary is not self-contained — users need `wl-clipboard`, `xclip`, or `xsel` installed (usually true on desktop distros, sometimes absent on minimal server images). The error path detects this and prints a clear install hint. This is documented in the README.

**Options Considered:**

- **Native everywhere, including X11/Wayland from C#.** Rejected — weeks of implementation for no observable user win over `xclip`. If Windows is the gap-fill platform anyway, spending our effort there is the right priority.
- **Shell-out everywhere, including Windows.** There's no standard Windows CLI clipboard tool to shell to (PowerShell's `Get-Clipboard` is slow to spin up and not always on PATH). P/Invoke is cheaper and faster.
- **Mixed: shell-out on Linux, native NSPasteboard on macOS via Objective-C runtime.** Considered but NSPasteboard adds complexity only paid off by rich-type support (HTML, images) — that's v2 work. For v1 text-only, `pbcopy`/`pbpaste` are fine.

## 2. Mode Autodetection with Explicit Override Flags

**Context:** `clip` is a single binary that both reads and writes the clipboard. How does the user express which they want?

**Decision:** Autodetect via `Console.IsInputRedirected` — if stdin is redirected, copy; else paste. Add explicit `-c`/`--copy` and `-p`/`--paste` flags to force either mode regardless of stdin state.

**Rationale:** The one-keystroke ergonomic (`clip` at a prompt to paste, `cmd | clip` to copy) is the killer feature for interactive use. Autodetection handles 99% of cases correctly. But scripts and cron jobs sometimes have stdin redirected from something they don't want treated as clipboard input (wrapper scripts inheriting a redirected stdin; pipelines where earlier stages produced no output but still show as redirected). Explicit `-c`/`-p` overrides are the escape hatch for those cases. This matches `xclip -i`/`-o` and `wl-copy`/`wl-paste` in spirit — a single tool that can go either way.

**Trade-offs Accepted:** Two ways to express intent (autodetect vs. explicit flag) adds documentation surface. Mitigated by making autodetect the common case and flags the edge-case override.

**Options Considered:**

- **Pure autodetect, no override flags.** Rejected — too fragile for scripts; no escape hatch when the user knows better than the heuristic.
- **Subcommands (`clip copy`, `clip paste`, `clip clear`).** Rejected — breaks the "no args = paste" ergonomic, which is the whole point of having a single binary rather than two `pbcopy`/`pbpaste`-style tools. Flag-based `--clear` is consistent with the rest of the suite (`retry --times`, `squeeze --decompress`).
- **Split into two binaries (`clipr` / `clipw`).** Rejected — breaks cross-suite composability (`pass --copy` expects one tool called `clip`) and doubles the install/distribution footprint.

## 3. Strip Trailing Newline on Paste by Default

**Context:** Users piping into `clip` often produce content with a trailing newline (`echo foo | clip` stores `foo\n`). What should paste output?

**Decision:** Strip exactly one trailing `\n` or `\r\n` by default. Offer `--raw`/`-r` to preserve every byte unchanged.

**Rationale:** Matches `$(…)` shell-substitution behaviour — users expect `$(clip)` to work like `$(cat file)`. Matches `wl-paste --no-newline`, which many users already rely on. Also works out well when composing: `clip | something` rarely wants a phantom trailing newline turning into an empty line. `--raw` exists for the precise-bytes case (binary-adjacent content, whitespace-sensitive transforms).

**Trade-offs Accepted:** Users who genuinely want the trailing newline must add `--raw`. Acceptable — the common case wins the short form.

**Options Considered:**

- **Always preserve (like `pbpaste`).** Rejected — mismatches shell `$(…)` conventions and forces every script to `printf '%s' "$(clip | sed 's/\n$//')"` or similar.
- **Strip all trailing whitespace.** Rejected — too aggressive; people do paste intentional whitespace.
- **Only strip the newline we added during copy (i.e. track provenance).** Rejected — requires state we don't have across invocations; violates "bridge, not a manager".

## 4. Empty Clipboard on Paste Returns Exit 0 with Empty Stdout

**Context:** What should `clip` do when asked to paste but the clipboard is empty?

**Decision:** Exit 0, write nothing to stdout, nothing to stderr. Match `pbpaste`.

**Rationale:** Empty is not an error. Scripts commonly run `clip` defensively and branch on content length (`content=$(clip); [ -n "$content" ] && ...`). An exit-1 would force every caller to wrap in `|| true`, which both obscures real errors and makes the common pattern uglier. `pbpaste` and modern `wl-paste` agree.

**Trade-offs Accepted:** Callers can't distinguish "empty clipboard" from "clipboard contains empty string" — but practically these are the same state and no sensible script action differs between them.

**Options Considered:**

- **Exit 1 on empty (grep-style).** Rejected — empty clipboard is not an error condition, and conflating it with one causes script-friction for the common case.
- **Exit 1 but with a flag like `--allow-empty` to opt out.** Rejected — over-engineering for a case that's better solved by having the right default.

## 5. Keep the Name `clip` Despite Collision with `System32\clip.exe`

**Context:** Windows ships `clip.exe` in `System32`. Our tool has the same name. Should we rename?

**Decision:** Keep the name `clip`. Document the `PATH`-ordering requirement in the README. Accept that Scoop users get our tool automatically; other install paths may need PATH attention.

**Rationale:** The Winix suite has already accepted this pattern for `less`, `man`, `nc`, and `schedule`. Our `clip.exe` is a strict superset of the Windows one — every `echo foo | clip` script continues to work unchanged, because the new behaviour (paste, clear) requires explicit invocation and never silently changes the write path. macOS and Linux have no `clip` at all, so a rename would cost three platforms to solve a cosmetic problem on one.

**Trade-offs Accepted:** On Windows, users who install our tool to a late-PATH location still hit the system `clip.exe`. Documented. Future `winix doctor`-style command could detect and warn.

**Options Considered:**

- **Rename to `clipx` (matching `treex`).** Rejected — ugly, and breaks the "one obvious name" expectation for the primary clipboard tool on Linux/macOS.
- **Rename to `wclip` or `cbrd`.** Rejected — same reason, with worse names.
- **Ship a multi-call binary that registers as both `clip` and something else.** Rejected — complexity without benefit.

## 6. Defer `--format text|html|rtf` to v2

**Context:** Should v1 support content-type selection (copy HTML to paste into Word with formatting preserved, etc.)?

**Decision:** Defer `--format` — and all rich-type support (images, files, `--list-formats`) — to v2.

**Rationale:** `--format` support is asymmetric across platforms:

- **Linux:** trivial (`xclip -t text/html`, `wl-copy --type text/html`).
- **Windows:** doable via P/Invoke with `CF_HTML` and its quirky `Version:0.9\r\nStartHTML:…` header format.
- **macOS:** `pbcopy` and `pbpaste` only handle plain text. Setting `public.html` on NSPasteboard requires either `osascript` (brittle) or native Objective-C runtime bindings — which is the exact work already deferred for image/file support.

Shipping `--format` in v1 would either ship macOS broken for non-text or drag NSPasteboard native work forward, splitting rich-type implementation across versions for no gain. v2 tackles rich-types as one coherent feature, including the macOS native backend.

**Trade-offs Accepted:** v1 ships text-only. Users who want HTML/RTF on the clipboard from the CLI must wait for v2 or use platform-specific tooling in the interim.

**Options Considered:**

- **Include `--format` in v1, macOS text-only.** Rejected — users would hit "works on Linux and Windows, silently degrades on macOS" surprise bugs.
- **Include `--format` in v1, with `osascript` on macOS.** Rejected — `osascript` invocation for rich-type clipboard data is brittle (escaping, large-payload handling) and would need to be rewritten during the v2 NSPasteboard work anyway.

## 7. `--primary` as a Linux-Only Flag (Silently Ignored Elsewhere)

**Context:** X11 and Wayland expose two selections: `CLIPBOARD` and `PRIMARY`. Windows and macOS have no such concept. How do we expose this?

**Decision:** Add `--primary` flag. On Linux, it targets the `PRIMARY` selection. On Windows and macOS, it is accepted and silently ignored. Documented.

**Rationale:** Users and scripts that need `PRIMARY` on Linux want an obvious way to ask for it. Users on Windows and macOS running the same script don't want it to fail — silent ignore makes the same script portable. "Accept and ignore" beats "error on platforms where this is meaningless".

**Trade-offs Accepted:** A flag that does nothing on some platforms can surprise users who assume failure means feature-gap. Mitigated by clear documentation.

**Options Considered:**

- **Error on non-Linux platforms when `--primary` is used.** Rejected — breaks portability of Linux scripts that get copied to a Mac.
- **Not supporting primary at all.** Rejected — a real Linux use case; trivial to implement via helper arg-vector changes.

## 8. Windows Retry Loop for `OpenClipboard`

**Context:** Windows clipboard can be briefly held by another process (Office, browsers). `OpenClipboard` returns false in that case.

**Decision:** Retry `OpenClipboard` up to 5 times at 50 ms intervals (total max wait ~250 ms). On exhaustion, exit 126 with `"clip: clipboard busy (another process holds it)"`.

**Rationale:** Brief clipboard contention is common and recoverable. 250 ms is imperceptible for CLI use and enough for 99% of contention cases. Beyond that, we surface a clear error rather than hanging.

**Trade-offs Accepted:** Adds up to 250 ms latency on contended opens (rare). Fixed upper bound prevents indefinite hangs.

**Options Considered:**

- **No retry.** Rejected — intermittent failures on otherwise-working systems are a bad first impression.
- **Longer retry with backoff.** Rejected — 250 ms total is already more than enough; longer feels sluggish for a command-line tool.

## Decisions Explicitly Deferred

| Topic | Why deferred |
|---|---|
| `--format text|html|rtf` | Rich-types bundled as v2 (see ADR §6). macOS needs NSPasteboard native work regardless. |
| `--image` / `--files` rich-type support | Same bucket as `--format`; land together in v2. |
| `--list-formats` | Couples with rich-types; v2. |
| Native NSPasteboard backend on macOS | Only pays off once we do rich-types; v2. |
| Native X11/Wayland backend on Linux | Weeks of protocol work for marginal benefit. Door left open. |
| Clipboard history | Explicitly not a goal — `clip` is a bridge, not a clipboard manager. |
| Windows `PATH`-ordering validation | Out of scope for `clip`; candidate for a future `winix doctor` check. |
| CRLF line-ending normalisation on Windows copy | Modern apps handle LF fine; forcing CRLF would surprise users. Revisit only if reports come in. |
