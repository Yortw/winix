# url — Architecture Decision Record

**Date:** 2026-04-20
**Status:** Accepted
**Companion document:** [2026-04-20-url-design.md](2026-04-20-url-design.md)
**Context:** Designing `url`, the third and final tool in Winix v0.4.0. A subcommand-based CLI for URL encode/decode/parse/build/join/query-edit.

---

## D1. Ship seven subcommands (`encode`, `decode`, `parse`, `build`, `join`, `query {get,set,delete}`)

**Context.** The pre-brainstorm notes listed `encode`, `decode`, `parse`, and `query {get,set,delete}`. During brainstorming we expanded the surface by two: `build` and `join`.

**Decision.** Ship seven subcommands in v1. `url list` (query enumeration) and `--user`/`--pass` on `build` are deferred.

**Rationale for `build`:** the first instinct was "shell printf handles URL assembly" — but that ignores per-value encoding, which is genuinely painful cross-shell. cmd.exe has no clean subprocess-substitution, so each `url encode "$value"` subshell is a `for /F` incantation. PowerShell has `[uri]::EscapeDataString` but Bash doesn't. Fish has `string escape --style=url` only in recent versions. Calling `url encode` N times for N query params also costs N × AOT-startup ≈ 3s for 10 params vs one `url build` call. Real gap; worth a subcommand.

**Rationale for `join`:** RFC 3986 §5 relative-URL resolution handles dot-segments, absolute relatives overriding base, query-only refs (`?page=2`), fragment-only refs (`#section`), and protocol-relative URLs (`//other.com/foo`). None of these can be done correctly with shell string concat. `new Uri(base, relative)` is one call.

**Trade-offs accepted.** Larger v1 surface than the original "trivial paper-cut" framing. Two extra commands and two extra pure-logic files (`UrlBuilder`, `UrlJoiner`). Offset: both are small (~60 lines each), both have high cross-platform value, and neither touches the crypto or security surface so the blast radius of bugs is low.

**Options considered.**
- Ship only the four original subcommands — rejected; builds shell scripts remain broken on cmd and awkward elsewhere.
- Also ship `url list` (query enumerate) — rejected as redundant with `parse --json | jq`.
- Also ship `build --user / --pass` — rejected; rare usage, security-discouraged.

## D2. Component encoding is the default for `url encode`

**Context.** URL percent-encoding is context-sensitive. The "right" set of characters to encode depends on whether the string is a path segment, a query value, a fragment, or a form field.

**Decision.** Default `--mode component` (`Uri.EscapeDataString` — RFC 3986 unreserved set, space → `%20`). Add `--mode {component,path,query,form}` and `--form` shorthand for the remaining variants.

**Rationale.** Component encoding matches JavaScript's `encodeURIComponent`, which is the mental model most web developers carry. It's the safest default — you can drop the output into any URL component without further escaping. Only users who know they need path-segment preservation (`/` unencoded) or form-data `+`-for-space need to reach for the flag.

**Trade-offs accepted.** "Safest default" encodes slightly more than strictly necessary in some contexts (e.g., `%2F` in a path segment is valid and intentional, but maybe unexpected to someone who expected `/` preserved). Documented in README; `--mode path` is the escape hatch.

**Options considered.**
- Default `--mode form` — rejected; form-encoding's `+`-for-space convention surprises users who want a generic URL-safe encoding.
- No default mode; force `--mode X` — rejected as hostile UX for the 95% case.

## D3. RFC 3986 decoding by default; `--form` opt-in for `+`-for-space

**Context.** Decoding behaviour differs between RFC 3986 (`+` is literal) and `application/x-www-form-urlencoded` (`+` means space). Applying the wrong one corrupts data.

**Decision.** Default `url decode` follows RFC 3986 (literal `+`). `--form` (or `--mode form`) flips to form-encoded semantics.

**Rationale.** If you give `url decode` an opaque URL component (e.g. a base64-encoded value in a query param), treating `+` as space silently corrupts the payload. RFC 3986 semantics preserve the input faithfully. Form semantics are opt-in because the user knows (or should know) when they're handling form-encoded data.

**Trade-offs accepted.** A user naively pasting a full form-encoded query string and running `url decode` on it will see literal `+` in the output and have to re-run with `--form`. One-flag friction; the alternative (silent corruption) is worse.

**Options considered.**
- Default `--form` — rejected; silent corruption risk.
- Auto-detect — rejected; no reliable heuristic.

## D4. `url parse` output: plain `key=value` lines by default, `--json` opt-in, `--field NAME` for single-value extraction

**Context.** Three reasonable shapes for structured output: JSON-only, plain-by-default with JSON opt-in, or plain with a field selector.

**Decision.** Plain `key=value` lines by default; `--json` for structured output (array-of-objects for `query` to preserve duplicates); `--field NAME` shortcut for single-value extraction.

**Rationale.** Matches the rest of the suite (`ids`, `digest`, `notify`, every other winix tool) — default plain, JSON opt-in. The `--field NAME` shortcut solves the "I just want the host" case without needing `jq`: `HOST=$(url parse "$u" --field host)` is one line with no extra tools. Critical for the cmd.exe user who doesn't have `jq`.

**Trade-offs accepted.** `query` as a raw string in plain mode means shell users who want to enumerate query keys need either `--json` + `jq` or `url query get`. Acceptable — duplicate keys and dotted key names would break any exploded plain-text format.

**Options considered.**
- JSON-only — rejected; forces `jq` dependency for trivial extraction.
- Plain with exploded `query.KEY=VALUE` lines — rejected; breaks on duplicate keys and dots in key names.

## D5. Per-subcommand normalisation rather than global default

**Context.** .NET's `Uri` class normalises by default (lowercase scheme/host, strip default ports, resolve dot-segments, decode unreserved escapes, IDN → Punycode). Normalisation is useful for canonicalisation but sometimes destructive (routing systems that care about case or dot-segments break).

**Decision.** Split behaviour by subcommand:

| Subcommand | Normalised? |
|---|---|
| `encode` / `decode` | N/A — string-level. |
| `parse` | No — reflect input faithfully. |
| `build` | Yes (unless `--raw`). |
| `join` | Yes (unless `--raw`). |
| `query set/delete` | Yes (unless `--raw`). |
| `query get` | N/A — returns a value, not a URL. |

**Rationale.** `parse`'s job is "show me what's there" — silently lowercasing the host would be a bug. `build`, `join`, and query-edits produce *new* URLs, so canonical output is the right default. `--raw` gives the escape hatch anywhere normalisation applies.

**Trade-offs accepted.** Users might be confused that `parse` preserves `HTTPS://EXAMPLE.COM:443/` literally but `build --scheme HTTPS --host EXAMPLE.COM --port 443` emits `https://example.com/`. Documented in README.

**Options considered.**
- Normalise everywhere by default — rejected; `parse` would silently transform input.
- Never normalise (force `--normalise` opt-in) — rejected; awkward for the common case of `build` producing canonical URLs.

## D6. Subcommand name `join` (not `combine` or `resolve`)

**Context.** Python's `urllib.parse.urljoin`, JavaScript's `new URL(relative, base)`, Go's `url.ResolveReference`, and .NET's `Uri` constructor all express the same RFC 3986 §5 resolution operation under different names.

**Decision.** Name the subcommand `url join`.

**Rationale.** Python's `urljoin` is the most widely-used reference implementation (scripters have muscle memory for it). `join` is shorter than `combine` and more discoverable than `resolve` (which requires knowing RFC 3986 terminology). Python and Node-familiar users will guess `join` first.

**Trade-offs accepted.** `resolve` is the technically correct RFC term, so purists may be briefly surprised. README documents the equivalence.

**Options considered.**
- `url combine` — rejected; no cross-language precedent.
- `url resolve` — rejected; requires RFC vocabulary, less discoverable.

---

## D7. No `Winix.Codec` additions

**Context.** Previous v0.4.0 tools (digest) added Hex/Base64/ConstantTimeCompare to the shared `Winix.Codec` library. `url` uses similar encode/decode primitives conceptually but operates at the URL level.

**Decision.** No `Winix.Codec` additions. All URL encoding lives in `src/Winix.Url/Encoder.cs` and `src/Winix.Url/Decoder.cs`.

**Rationale.** URL encoding is a thin wrapper around `Uri.EscapeDataString` plus mode-specific logic (path-segment preservation, `+`-space swap). It's not a reusable primitive in the sense that Hex or Base64 are — no other current or planned Winix tool needs URL-specific encoding. Lifting it into `Winix.Codec` would add a dependency without demand.

**Trade-offs accepted.** If a future tool does need URL encoding (e.g., a `qr` tool encoding a Wi-Fi network URI), we'd duplicate some code. Acceptable — trivial to promote later if needed.

**Options considered.**
- Add `Winix.Codec.UrlEncode` — rejected; premature abstraction.

## Decisions Explicitly Deferred

| Topic | Why deferred |
|---|---|
| `url query get --all` (return all values for duplicate key) | First-match covers most use cases; straightforward v2 addition. |
| `url query list` (enumerate all keys/values) | Redundant with `parse --json` pipeline. |
| `url build --user / --pass` (userinfo on build) | Rare usage; embedding credentials in URLs is security-discouraged. **Note:** userinfo *is* preserved by `query set/delete` via an internal `UrlBuilder.Build(userInfo:)` parameter — that's data-loss protection, not a user-visible `build` flag. |
| Batch mode (multi-URL input) | Shell loops cover it; keeping CLI one-URL-per-invocation simplifies the model. |
| HTML entity encode/decode | Different encoding domain; out of scope for a URL-focused tool. |
| IDN → Punycode conversion | Design initially claimed this, but `System.Uri.AbsoluteUri` preserves Unicode hosts (doesn't auto-Punycode). Shipping as "Unicode preserved" for v1 — applications needing Punycode can pass through `Uri.IdnHost` externally. Revisit with an opt-in `--idn` flag in v2. |
| Stdin input (`-` positional) | Design initially promised `echo "x" \| url encode -`. Not implemented in v1. A bare `-` is currently encoded as the literal character. Design doc and README updated to reflect v2 status. |
