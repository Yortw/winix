# ADR: files — Cross-Platform File Finder

**Date:** 2026-03-31
**Status:** Proposed
**Context:** Winix needs a `find` replacement that works natively on Windows and improves on `find`'s hostile expression syntax. Also serves as the proving ground for `--describe` AI discoverability.
**Related:** [files Design](2026-03-31-files-design.md), [AI Discoverability Design](2026-03-31-ai-discoverability-design.md), [CLI Conventions](2026-03-29-winix-cli-conventions.md)

---

## Decision 1: Find everything by default

**Context:** `fd` (Rust) hides `.gitignore`'d and hidden files by default, targeting developers in git repos. `find` (POSIX) shows everything. `files` needs a default behaviour.

**Decision:** Show all files by default. `--gitignore` opts into gitignore filtering. `--no-hidden` opts into hiding dotfiles.

**Rationale:** The tool's job is to find files — hiding results by default means silent incompleteness. AI agents frequently need hidden files (`.env`, `.github/`, `.editorconfig`) and build artifacts. Developers searching `bin/` or `node_modules/` for dependency investigation need everything visible. Users who want fd-style defaults can use `--gitignore --no-hidden`.

**Trade-offs Accepted:** Noisier default output in large repos with `node_modules/` etc. Acceptable — filtering flags exist, and completeness is the safer default.

**Options Considered:**
- **fd-style defaults (hide ignored + hidden):** Rejected — makes the tool's output incomplete by default, hostile to AI agents and non-dev use cases.
- **Configurable default via env var:** Rejected — invisible state makes behaviour unpredictable.

## Decision 2: Shared Winix.FileWalk library from day one

**Context:** `treex` (directory tree viewer) will also need directory walking, glob matching, and predicate filtering. The walking engine could live in `Winix.Files` and be extracted later, or be a shared library from the start.

**Decision:** Create `Winix.FileWalk` as a separate shared library. `Winix.Files` depends on it for walking, adds output formatting. `treex` will reference the same library when built.

**Rationale:** The walking engine's shape is predictable: enumerate + filter + yield. Both consumers (flat file list vs tree view) need the same walking and filtering, just different rendering. Building the shared library now avoids a refactoring pass when treex starts.

**Trade-offs Accepted:** Designing a shared API before we have two consumers risks minor over-generalisation. Mitigated by keeping the API narrow (FileWalker + FileEntry + options record).

**Options Considered:**
- **Build in Winix.Files, extract later:** Rejected — the extraction refactor would touch files tool internals for no benefit, and we know treex is next.
- **Everything in one library:** Rejected — the walking engine and the output formatting are separate concerns with different consumers.

## Decision 3: GitIgnoreFilter in ShellKit, wrapping `git check-ignore`

**Context:** Gitignore rules are complex: nested `.gitignore` files, `.git/info/exclude`, global excludes (`core.excludesFile`), negation patterns, directory-only patterns. Peep's file watcher already shells out to git for this.

**Decision:** Add `GitIgnoreFilter` to ShellKit. Implementation wraps `git check-ignore --stdin -z` for correctness. Interface is `Create(rootPath)` → `IsIgnored(relativePath)`. Returns null if not in a git repo.

**Rationale:** `git check-ignore` handles all the edge cases correctly — reimplementing gitignore parsing is a known rabbit hole (nested overrides, negation priority, global config). The wrapper interface is clean enough that a native implementation can replace it later without consumers changing. ShellKit is the right home because Peep's FileWatcher can also use it.

**Trade-offs Accepted:**
- Requires `git` on PATH. Acceptable — `--gitignore` is opt-in, and anyone using gitignore filtering has git installed.
- Process spawning overhead. Mitigated by `--stdin` mode (one long-running process, batched input).

**Options Considered:**
- **Native C# gitignore parser:** Rejected for v1 — high implementation effort, high bug risk. Can be added later behind the same interface.
- **NuGet library (e.g. LibGit2Sharp):** Rejected — heavy native dependency, AOT-hostile, overkill for ignore checking.

## Decision 4: Both glob and regex, complementary

**Context:** `find` uses glob patterns (`-name '*.cs'`). `fd` uses regex by default. Users expect different things.

**Decision:** Support both `--glob` and `--regex` flags. Multiple patterns of the same type are OR'd. When both types are present, they AND (must match at least one glob AND at least one regex). Both match against filename only.

**Rationale:** Glob covers 95% of use cases and is what AI agents naturally produce. Regex adds power for the remaining 5% (e.g. `test_\d+\.cs`). Making them complementary (not mutually exclusive) enables `--glob '*.cs' --regex 'test_\d+'` — a useful combination. Implementation cost is trivial (one `Regex.IsMatch` call).

**Trade-offs Accepted:** Two pattern systems could confuse users. Mitigated by: glob is the primary/obvious choice, regex is the power-user escape hatch.

**Options Considered:**
- **Glob only:** Considered — simpler interface. Rejected because regex support is trivial to add and valuable for edge cases.
- **Regex only (fd-style):** Rejected — glob is more natural for filenames and more predictable for AI agents.
- **Mutually exclusive:** Rejected — no reason to forbid combining them.

## Decision 5: No `--compat` flag

**Context:** Winix CLI conventions include a `--compat` flag for tools that replace a specific predecessor (e.g. squeeze → gzip). `files` replaces `find`.

**Decision:** No `--compat` flag. No compatibility aliases for `find`'s flags.

**Rationale:** `find`'s syntax is expression-based (`-name`, `-exec {}\;`, `-mtime +7`), not flag-based. A `--compat` flag would need to implement a fundamentally different parsing model. The overlap between `find`'s CLI surface and `files`'s flag-based approach is too small for compatibility to be meaningful. Users moving from `find` need to learn `files`'s flags regardless.

**Trade-offs Accepted:** No muscle-memory bridge for `find` users. Acceptable — the syntax is so different that aliases would be misleading.

## Decision 6: `--long` for human-readable detail

**Context:** The default output (paths only) is optimised for piping. NDJSON provides structured data. There's a gap for humans who want to see file details without full JSON.

**Decision:** `--long` / `-l` flag produces tab-delimited columns: path, size (bytes with commas), modified (local time, no seconds), type.

**Rationale:** Sits between paths-only (piping) and NDJSON (structured). Tab-delimited is both human-readable and `cut`/`awk`-parseable. Mirrors the `ls -l` convention that developers already know. Size in exact bytes with commas is more useful than human-rounded sizes (2.3K) — no ambiguity about what the number means.

**Trade-offs Accepted:** Another output format to maintain. Acceptable — it's a simple formatting function.

## Decision 7: `files` as the AI discoverability proving ground

**Context:** The [AI discoverability design](2026-03-31-ai-discoverability-design.md) defines `--describe`, `llms.txt`, and AI guide docs. These need to be built and proven before retrofitting to existing tools.

**Decision:** `files` is the first tool to ship with `--describe` metadata, an AI guide (`docs/ai/files.md`), and `llms.txt`. After proving the patterns, retrofit to timeit, squeeze, peep, wargs.

**Rationale:** Greenfield tool — we can iterate on the `--describe` format without backwards compatibility concerns. If the format needs adjusting, we adjust before it's on four other tools.

## Decision 8: `--ext` extension shorthand

**Context:** The most common file-finding use case is filtering by extension (`*.cs`, `*.json`). `--glob '*.cs'` works but is verbose for this common case. `fd` has `--extension` and it's one of its most-used flags.

**Decision:** Add `--ext EXT` as sugar for `--glob '*.<ext>'`. Repeatable (OR logic). No leading dot expected (`--ext cs` not `--ext .cs`; leading dot stripped with a warning if provided).

**Rationale:** High-frequency use case deserves a short flag. Trivial to implement — internally generates a glob pattern. Repeatable lets you do `--ext cs --ext fs` to match multiple extensions.

**Trade-offs Accepted:** One more flag in the interface. Acceptable — it reduces the most common invocation from `--glob '*.cs'` to `--ext cs`.

---

## Decision 9: Platform-aware case sensitivity

**Context:** NTFS (Windows) and APFS (macOS) are case-insensitive by default. ext4/xfs (Linux) are case-sensitive. A pattern like `--glob '*.CS'` should match `file.cs` on Windows/macOS but not on Linux — matching the filesystem's own behaviour.

**Decision:** Default case sensitivity matches the platform's filesystem: case-insensitive on Windows and macOS, case-sensitive on Linux. Override with `--ignore-case` / `--case-sensitive`.

**Rationale:** Patterns should behave consistently with the filesystem — if `cd myDir` and `cd MYDIR` reach the same directory, then `--glob 'mydir*'` and `--glob 'MYDIR*'` should match the same files. This is the least-surprising default. Override flags handle edge cases (case-sensitive APFS volumes, files copied from Windows to Linux).

**Trade-offs Accepted:** macOS users with case-sensitive APFS volumes get the wrong default. Acceptable — case-sensitive macOS volumes are rare, and `--case-sensitive` is one flag away.

**Options Considered:**
- **Smart-case (fd-style):** Case-insensitive unless pattern contains uppercase. Rejected — clever but surprising. Users shouldn't need to know about smart-case to predict behaviour.
- **Always case-sensitive:** Rejected — hostile on Windows/macOS where the filesystem itself is case-insensitive.
- **Always case-insensitive:** Rejected — would match files that the filesystem distinguishes on Linux.

## Decision 10: Text vs binary content detection

**Context:** AI coding agents frequently need to find files they can read/analyze. No file finder distinguishes text from binary as a built-in predicate — users resort to `file --mime-type` piping or manual extension filtering (which misses extensionless files and misnamed files).

**Decision:** Add `--text` and `--binary` flags. Detection uses the same null-byte heuristic as git: read the first 8KB of each candidate file; if null bytes are present, it's binary. Mutually exclusive. Applied as a late filter after other predicates have narrowed the result set, minimising I/O overhead.

**Rationale:** Directly serves the AI-friendly tool thesis — an agent can `files . --text --ext cs` or just `files . --text` to find readable files. Also useful for developers ("find binary files that shouldn't be committed"). The null-byte heuristic is battle-tested (git has used it for decades) and trivial to implement.

**Trade-offs Accepted:**
- Reads 8KB per candidate file — I/O overhead for large result sets. Mitigated by applying after other predicates narrow the set.
- Heuristic isn't perfect — a text file containing a null byte (rare but possible) would be misclassified as binary. Acceptable — same limitation as git.

**Options Considered:**
- **MIME type detection (libmagic):** Rejected — heavy native dependency, AOT-hostile, overkill for a binary/text distinction.
- **Extension-based guessing:** Rejected — misses extensionless files, wrong for misnamed files, requires maintaining a list.

## Decisions Explicitly Deferred

| Topic | Why deferred |
|-------|-------------|
| Sorting of output | Filesystem order is fastest and matches `find` behaviour. Sorting options (`--sort name`, `--sort size`, `--sort modified`) can be added in v2 if needed. |
| `--exec` flag | `find -exec` is powerful but complex. `files \| wargs` achieves the same thing with standard pipe composition. No need to duplicate wargs's job inside files. |
| Full-path matching for patterns | v1 matches patterns against filename only. Full-path matching (`--path-glob`, `--path-regex`) can be added later if needed. |
| Native gitignore parser | v1 wraps `git check-ignore`. Native parsing can replace it later behind the same `GitIgnoreFilter` interface. |
| Treex integration | `Winix.FileWalk` is designed for sharing, but treex design is a separate effort. |
| `--count` flag | `files ... \| wc -l` or `files --ndjson ... \| jq -s length`. Not worth a dedicated flag in v1. |
| Parallel directory walking | v1 uses single-threaded enumeration. Parallel walking (like fd) can yield 2-5x speedup on SSD/NVMe for large trees. Deferred to v2 — the `FileWalker` interface supports this as an internal optimization without API changes. |
