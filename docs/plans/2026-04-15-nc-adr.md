# nc — Architecture Decision Record

**Date:** 2026-04-15
**Status:** Accepted
**Companion design:** [2026-04-15-nc-design.md](2026-04-15-nc-design.md)

## Context

Winix is missing a netcat-style tool for TCP/UDP send/receive and port checks. Windows users currently fall back to `Test-NetConnection` (verbose) or fragile PowerShell snippets. Real netcat exists on Linux/macOS but in incompatible forks (BSD, GNU, ncat) with subtly different flags and behaviour. We want a single cross-platform tool that fits the Winix philosophy: do one thing well, ship a small AOT binary, prefer descriptive flags but accept netcat muscle memory.

This ADR captures the design decisions made during brainstorming on 2026-04-15. v0.2.0 is the target release.

## Decisions

### 1. Feature scope — "client + listener" tier, not full ncat

**Context:** Real netcat has dozens of features, many legacy or niche. Building all of them is a multi-week effort that delays v0.2.0 and adds long-term maintenance burden for features almost nobody uses.

**Decision:** v0.2.0 ships TCP/UDP client, TCP/UDP single-connection listener, port-open check (with single/range/list/mixed syntax), TLS for client connections, IPv4/IPv6 selection, timeouts, file transfer (free via stdin/stdout). Skips hex dump, source-port binding, exec-on-connect, SOCKS, port forwarding, multi-connection listen, server-side TLS, DTLS.

**Rationale:** This covers ~95% of real-world netcat use for the Winix audience (Windows devs/sysadmins doing "is this thing up?" diagnostics, smoke tests, service development). The skipped features are either (a) better served by other tools (Wireshark for protocol debugging, SSH for tunnelling, socat for port forwarding), (b) security risks (`-e`), or (c) legacy edge cases.

**Trade-offs accepted:** Power users coming from ncat may find specific niche features missing. Issue-driven additions are fine; speculative inclusion is not.

**Options considered:**
- Minimal "port poker" only — too narrow; a netcat tool that can't even send a string is not actually netcat.
- Full ncat parity — too large; would slip v0.2.0 by weeks and add features we don't believe most users need.

### 2. CLI surface — Winix long flags primary, netcat short flags as aliases

**Context:** Existing nc users have muscle memory for short flags (`-z`, `-l`, `-u`, `-w`, `-4`, `-6`). Existing Winix tools (`whoholds`, `files`, `treex`, `schedule`) use descriptive long flags as the primary form for discoverability. We can't have both as the canonical form.

**Decision:** Long-form flags are primary (`--listen`, `--check`, `--udp`, `--timeout`, `--ipv4`). Common netcat short flags are accepted as aliases. The canonical help/docs/examples lead with long-form.

**Rationale:** Internal consistency across the Winix suite matters — users learning Winix should see one CLI style, not eleven. Muscle memory is preserved for the people who care about it. The cost is one extra entry per flag in the parser builder.

**Trade-offs accepted:** Examples in docs read slightly more verbosely than classic netcat tutorials. Quick scripts may use either form.

**Options considered:**
- Faithful netcat (short flags primary) — breaks internal Winix consistency for one tool.
- Subcommand model (`nc connect HOST PORT`) — breaks classic `nc HOST PORT` muscle memory for everyone.

### 3. TLS — client only (v0.2.0)

**Context:** Modern services are mostly TLS. A netcat without TLS is increasingly limited. Adding TLS for clients via `SslStream` is small and AOT-compatible. Adding TLS for listen mode requires server certificate management (generation, loading, storage) which is a separate design problem.

**Decision:** v0.2.0 supports `--tls` only in client modes (default connect). `--insecure` skips cert validation with a stderr warning. Server-side TLS deferred.

**Rationale:** Client TLS unblocks the most common modern use case (HTTPS smoke tests, TLS-protected service checks) with minimal code. Server TLS is a follow-up that can wait until someone files an issue requesting it.

**Trade-offs accepted:** Users who want to test their own TLS server by running `nc --listen --tls 8443` will need to wait for a follow-up release.

**Options considered:**
- No TLS at all — would force users to drop to `openssl s_client` for half their use cases.
- Full TLS on both sides in v0.2.0 — adds cert-management surface area that doubles the design effort.

### 4. Half-close on stdin EOF (with `--no-shutdown` escape hatch)

**Context:** When stdin reaches EOF (typical for `echo "..." | nc target 80` or `nc target 80 < file`), the question is: should we close the write half of the socket so the peer knows we're done sending? Original netcat had problems with this (needed `-q` flag); modern ncat does it by default. A small minority of protocols (line-oriented chats with no logout) interpret half-close as a full close and drop the connection.

**Decision:** Default behaviour is to call `Socket.Shutdown(Send)` on stdin EOF. Provide `--no-shutdown` to suppress.

**Rationale:** The default is what 99% of users want — without it, the very common HTTP-like pattern (`echo request | nc | response`) hangs forever, which is a terrible first-run experience. The escape hatch covers the long tail.

**Trade-offs accepted:** Adds one more flag to the surface, but it's well-named and only needed by a tiny minority.

**Options considered:**
- Always shutdown (no escape hatch) — would surprise users with line-protocol services.
- Never shutdown (require explicit flag) — would surprise the 99% with the most common use case.

### 5. Port range syntax matches netcat (`80,443,8080-8090`)

**Context:** `--check` mode benefits from probing multiple ports in one invocation. Real netcat already has range syntax that users may know.

**Decision:** Accept single (`80`), list (`80,443,8080`), range (`80-100`), and mixed (`80-100,443,8080-8090`). Hard cap on total ports per invocation: TBD only if perf becomes an issue (initial: no cap).

**Rationale:** Matches netcat conventions. Implementation is a small parser. Useful for the "what's open on this DB server?" workflow without crossing into nmap territory (no service fingerprinting, no host ranges, single target only).

**Trade-offs accepted:** Adds a port-range parser and tests; outweighs the value of doing this with shell scripts.

**Options considered:**
- Single port only — leaves users writing `for port in {80..1024}; do nc -z ...; done` shell loops.
- nmap-style host + port ranges — out of scope; nmap is the right tool for that.

### 6. JSON to stderr, bytes to stdout (consistent with whoholds/timeit)

**Context:** Winix tools that emit summaries (`timeit`, `whoholds`) put `--json` output on stderr so it doesn't pollute stdout when used in pipelines.

**Decision:** Same convention here. `nc target 80 --json < req > resp` puts the response bytes in `resp` and the JSON summary on stderr where the user can see it (or redirect separately).

**Rationale:** Consistency across the suite. Users who need both can redirect stderr to a file; users who don't get clean piped data.

**Trade-offs accepted:** Mildly surprising for first-time users expecting JSON on stdout. Documented prominently in README.

### 7. No new shared library

**Context:** `whoholds` already exists and deals with networking. Could there be shared code?

**Decision:** No. `nc` lives entirely in `Winix.NetCat`. No code reuse with `Winix.WhoHolds`.

**Rationale:** The two tools share zero implementation surface despite both being "network tools." `whoholds` queries the local OS connection table via Win32 IP Helper API to find which local processes own a port. `nc` opens its own outbound/inbound sockets via `System.Net.Sockets.TcpClient`/`UdpClient`/`SslStream`. There's no abstraction to share.

**Trade-offs accepted:** None — this is just acknowledging that the tools are unrelated despite surface similarity.

## Decisions Explicitly Deferred

| Topic | Why deferred |
|-------|-------------|
| Hex dump (`-o`) | Niche; users with serious protocol-debugging needs reach for Wireshark/mitmproxy. Can add later if requested. |
| Source-port binding (`-p`) | Rarely needed in modern networks. Add later if requested. |
| Exec on connect (`-e`) | Security risk; modern netcats restrict or remove. Likely permanently out. |
| SOCKS proxy support (`-x`) | Niche; SSH `-D` covers most cases. Reconsider if multiple users ask. |
| Port forwarding / relay mode | That's `socat`'s job. Building a small `socat` is significant scope. |
| Multi-connection listen | Adds concurrency complexity. v0.2.0 ships single-connection only. |
| TLS for listen mode (server certs) | Requires cert management surface. Add when there's demand. |
| DTLS (TLS over UDP) | Niche. |
| `--parallel N` flag for check mode | Default of 32 concurrent should be fine for v0.2.0. Add knob if anyone needs it. |
| Configurable per-port timeout in check mode | Single `--timeout` covers it for now. |
