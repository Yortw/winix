# ADR — trash

**Date:** 2026-05-29
**Status:** Accepted
**Context:** Designing `trash`, a cross-platform safe-delete-to-recycle-bin CLI for the Winix
suite (v0.4.0 tool batch, after mksecret). Windows lacks a built-in recycle-bin CLI; Linux/macOS
have strong native tools (`trash-cli`, `macos-trash`) we must match rather than undercut.
**Design doc:** [2026-05-29-trash-design.md](2026-05-29-trash-design.md)

---

## Decision 1 — v1 scope: delete + list + empty; restore deferred

**Context.** The full shape (`trash`, `--list`, `--restore`, `--empty`) spans three OSes.
Delete-to-bin and empty are cheap everywhere; `--restore` (and the metadata it needs) is the
expensive, blast-radius piece — especially on Windows (recreate from `$R`/`$I` or Shell COM) and
macOS (no public API to read the Put-Back source).

**Decision.** v1 = `trash <paths>`, `--list` (read-only), `--empty`. `--restore` deferred to v2.

**Rationale.** Ships the high-value 80% (safe delete on the platform that lacks it, plus
visibility) without the cross-platform restore-indexing cost. List is cheap on Linux/Windows and
useful immediately.

**Trade-offs accepted.** Users restore via the OS GUI in v1. We're a deliberate subset of
`trash-cli` (which has restore) until v2.

**Options considered.** (a) Delete+empty only — too thin, `--list` is cheap and valuable.
(b) Full incl. restore — largest, highest-risk, not "small". (c) Asymmetric restore (some OSes) —
rejected for v1; revisit in v2 as one coherent feature.

---

## Decision 2 — macOS backend uses `NSFileManager.trashItem` (native Obj-C interop)

**Context.** macOS has no clean trash *syscall*. Options: direct move to `~/.Trash` (loses Finder
"Put Back"); shell out to `osascript`/Finder (needs Finder running, fails headless); or
`NSFileManager.trashItem` via Objective-C interop (native Put Back + per-volume routing, no
Finder). The respected `macos-trash` tool uses `trashItem`.

**Decision.** Use `NSFileManager.trashItem` via `objc_msgSend` P/Invoke in v1.

**Rationale.** It is the proven-correct native approach; a naive `~/.Trash` move would be "worse
than native" (broken Put Back), which is a hard sell for a cross-platform tool. `trashItem` also
gets cross-volume routing for free. The blocking concern was *testability* (Obj-C interop can't be
run on the Windows dev box) — resolved by the macOS CI runner (`ci.yml` matrix includes
`macos-latest`), which a `SkippableFact` integration test uses to verify the interop end-to-end.

**Trade-offs accepted.** First native Obj-C interop in the suite (no in-repo pattern to copy;
Keychain shells to `security`). ~100–150 LOC of `objc_msgSend`/selector/`NSError**` marshalling
that the author cannot run locally — mitigated by the mandatory macOS-CI integration test.

**Options considered.** Direct `~/.Trash` move — rejected (no Put Back; "worse than native").
`osascript`/Finder — rejected (headless failure, slower, inconsistent with the Linux direct-write
approach; modern tools moved away from it).

## Decision 3 — Linux implements the full FreeDesktop spec incl. multi-volume

**Context.** A simple move into `~/.local/share/Trash` fails (`EXDEV`) for files on other mounts.
The spec defines per-mount trash dirs (`$topdir/.Trash-$uid`). `trash-cli` implements this.

**Decision.** Implement the full spec, including multi-volume top-dir trash with the spec's
sticky-bit/symlink safety checks.

**Rationale.** "Cross-platform but worse than the native tool" is a hard sell. Matching
`trash-cli`'s capability (and interoperating via shared spec dirs — our `--list` sees GUI/CLI
file-manager-trashed items and vice-versa) is the whole reason to exist on Linux. (Windows and
macOS get multi-volume free via their APIs, so Linux is the only hand-rolled multi-volume path.)

**Trade-offs accepted.** +60–80 LOC and edge cases (mount detection, dir creation, permissions)
vs a home-volume-only subset.

**Options considered.** Home-volume-only (refuse cross-volume) — rejected: undercuts the native
tool. Copy-fallback across volumes — rejected: silently turns move into slow non-atomic copy,
changes semantics.

## Decision 4 — `--empty` is guarded: TTY prompt, `--yes` required non-interactively

**Context.** `--empty` is irreversible. A scripted/piped invocation with no guard risks an
accidental mass-permanent-delete.

**Decision.** Prompt `Permanently delete N item(s)? [y/N]` when stdin is a TTY; require `--yes`
to proceed when non-interactive.

**Rationale.** Safe-by-default for the destructive operation; matches the suite value of
minimising support calls. Trashing itself stays unprompted (it's recoverable).

**Trade-offs accepted.** Scripts must pass `--yes` explicitly (intended friction).

**Options considered.** Always require `--yes` — slightly safer but worse interactive UX.
No guard — rejected (irreversible).

---

## Decision 5 — flag-mode CLI, not subcommands

**Context.** The operations could be subcommands (`trash put/list/empty`) or mode flags on a
positional-taking command.

**Decision.** Flag-mode: bare positionals = trash; `--list`/`--empty` are mutually-exclusive mode
flags.

**Rationale.** Keeps the common case ergonomic (`trash file.txt`), matches the memory-captured
shape, and `trash-cli`'s separate-binary model maps naturally onto flags for a single binary.

**Trade-offs accepted.** Mode mutual-exclusion must be validated explicitly in the arg parser.

---

## Decision 6 — Windows `--list` parses `$I` metadata directly (no COM)

**Context.** Enumerating the recycle bin can be done via Shell COM (`IShellFolder`) or by parsing
the `$Recycle.Bin\<SID>\$I*` metadata files.

**Decision.** Parse `$I` files directly (v2 format).

**Rationale.** AOT-clean (COM interop under NativeAOT is painful), gives original path + deletion
time + size, and is unit-testable against captured byte fixtures.

**Trade-offs accepted.** Must handle format version + multi-drive + access-denied edge cases.

---

## Decisions Explicitly Deferred

| Topic | Why deferred |
|---|---|
| `--restore` | Cross-platform restore-indexing + Windows `$R`/`$I` recreation; ship v1 value first. v2. |
| macOS `--list` original paths | Inherent cost of native-Trash integration (Decision 2), **confirmed not a missed technique** (2026-05-29 research): the native `~/.Trash` keeps the original path in the binary `.DS_Store` (undocumented; only GUI apps parse it). `macos-trash` doesn't list; `trash-cli` lists original paths only by *bypassing* `~/.Trash` for its own `~/.local/share/Trash` store — losing Put Back + Finder integration (trash-cli issues #127/#284), the very trade-off we rejected. **v2 path:** write our own sidecar metadata (original path → trash filename) when *we* trash, keeping native `trashItem`; pairs with restore (which needs the original path). Finder-trashed items show "origin unknown". |
| macOS restore (Put-Back read) | Same private-store constraint; the hard part of v2 macOS restore. |
| `--empty` real-OS integration test | OS empty APIs are all-or-nothing → would wipe unrelated items. Deliberately gated: fake-backend + unit-level prompt/`--yes` + manual smoke. Documented ship-gate, not a silent skip. |
| Windows glob expansion | Suite-wide ShellKit gap; trash inherits the deferred fix. |
