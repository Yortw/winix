# ADR: ids Tool Design Decisions

**Date:** 2026-04-17
**Status:** Proposed
**Context:** Design for `ids`, an identifier-generator CLI for the Winix suite (UUID v4, UUID v7, ULID, NanoID).
**Related:** [ids design doc](2026-04-17-ids-design.md)

---

## 1. Default ID Type is UUID v7 (not v4)

**Context:** When a user runs `ids` with no `--type` flag, what do they get?

**Decision:** Default to UUID v7.

**Rationale:** UUID v7 is a strict upgrade over v4 for most use cases: it's time-ordered (generated IDs sort in creation order), keeps the hot pages hot when used as a database primary key, is canonically RFC 9562, and costs nothing to generate on .NET 10 (native `Guid.CreateVersion7()`). The v4 use case — "I just need a random identifier with no temporal signal" — is still available via `--type uuid4`. For the overwhelmingly common case of "I need an ID for a row in a table, a request trace, a file name", v7 is the better default.

**Trade-offs Accepted:** UUID v7 leaks approximate creation time (Unix ms, 48 bits). For the small subset of use cases where this is a problem (e.g. session tokens in a context where observers shouldn't infer creation time), users must explicitly pass `--type uuid4`. Documented.

**Options Considered:**

- **Default to `uuid4`.** Conservative, universally understood, matches what `uuidgen` gives you. Rejected because v7 is strictly better for the common case and the "no temporal signal" use case is rare enough to warrant an explicit opt-in.
- **Default to `ulid`.** Shorter (26 chars), time-ordered, URL-safe. Rejected because UUID is the lingua franca — users dropping `ids` into existing systems expect a UUID-shaped output by default. `--type ulid` is one flag away.
- **No default — require `--type`.** Rejected — breaks the "just give me an ID" ergonomic that makes this tool worth reaching for.

## 2. Create `Winix.Codec` Shared Library Now

**Context:** `ids` needs Crockford base32 encoding (for ULID) and a secure RNG wrapper. `digest` (next tool in Tier 1) will additionally need hex, base32, base64, constant-time byte comparison, and the same RNG. Should these land in a shared library now or inline in `Winix.Ids` and hoist later?

**Decision:** Create `Winix.Codec` as part of this work. Put Crockford base32 + `ISecureRandom` in it. `Winix.Ids` references it. `digest` will extend it when built.

**Rationale:** The `Winix.FileWalk` precedent is directly applicable: that library was created alongside `files` (its first consumer) and reused when `treex` was built. No rework needed at reuse time. Creating `Winix.Codec` now costs one extra `.csproj` and a one-line project reference; the alternative (inline the code, hoist later) guarantees a refactor when `digest` arrives and risks subtle API drift if the two consumers diverge in the interim.

**Trade-offs Accepted:** Slightly more scaffolding upfront. A new NuGet package ID to register (`Winix.Codec` — library-only, consumed transitively, not a standalone tool).

**Options Considered:**

- **Inline encoders inside `Winix.Ids`, hoist to `Winix.Codec` when `digest` lands.** Rejected — guaranteed rework, plus the hoist often happens under deadline pressure and leads to "good enough" API shapes that stick around.
- **Combine all encoding + crypto helpers into `Yort.ShellKit`.** Rejected — ShellKit is environment/formatting infrastructure (console detection, color, arg parsing). Codec/crypto is a distinct concern, and conflating them would grow ShellKit's surface in the wrong direction.

## 3. Format Shape and Case are Orthogonal Flags

**Context:** Users want various UUID output shapes — hyphenated, no-hyphens, braces, URN prefix — and sometimes uppercase. How do we expose this on the flag surface?

**Decision:** Named shape presets via `--format {default, hex, braces, urn}`, with a separate `--uppercase` boolean that applies to any shape.

**Rationale:** Shape and case are genuinely orthogonal concerns. Uppercase braced GUIDs are the Windows registry idiom (`{XXXXXXXX-…}`); uppercase hex shows up in firmware and hash manifests; lowercase hyphenated is the default in almost all documentation and APIs; URN-prefixed matches the RFC 9562 canonical form. Forcing shape + case into a single presets list would either multiply the enum (8 values instead of 4) or deny sensible combinations. Independent booleans (`--no-hyphens`, `--uppercase`, `--braces`) are flexible but multiply into eight permutations the user has to reason about and document. Named shape + orthogonal case is the sweet spot: four shapes × two cases = eight outputs, all discoverable, all named.

**Trade-offs Accepted:** Users need to learn two concepts rather than one. Mitigated by `--help` showing both side-by-side with examples.

**Options Considered:**

- **Single `--format` preset with shape + case combined** (e.g. `hex`, `hex-upper`, `braces`, `braces-upper`, …). Rejected — 8-value enums are hard to document and teach, and the combinatorial growth continues if we add new shapes later.
- **Independent booleans for shape (`--no-hyphens`, `--braces`, `--urn`) and case (`--uppercase`).** Rejected — users need to understand that `--no-hyphens` and `--braces` are mutually exclusive, and the help text surface grows. Named presets with error on unknown values communicate the mutual exclusion automatically.

## 4. Named NanoID Alphabet Presets, Not Arbitrary Strings (v1)

**Context:** NanoID is parameterised by alphabet + length. How flexible should the v1 flag surface be?

**Decision:** `--alphabet {url-safe, alphanum, hex, lower, upper}` — named presets only. Arbitrary custom alphabets deferred to v2.

**Rationale:** Named presets cover essentially all real-world uses:

- `url-safe` (default, 64 chars including `_-`) — URL/slug use, the NanoID primary use case.
- `alphanum` (62 chars, no punctuation) — file names, database keys where `_-` would cause issues.
- `hex` (16 chars) — short random hex tokens, a common bonus use case for NanoID.
- `lower` / `upper` (32 chars each with digits) — case-constrained systems (case-insensitive DB lookups, case-sensitive filesystems).

Custom alphabets open a validation rabbit hole (minimum size warnings, duplicate chars, UTF-8 multi-byte chars, help-text formatting of user-provided strings). All resolvable in v2 if a real need emerges. Presets also let us pick sizes that hit powers of two (16, 32, 64) so the rejection-sampling rate is zero.

**Trade-offs Accepted:** A user who wants (say) "digits only" today has to pick `alphanum` and post-filter, or pre-filter characters from the output. Rare ask; v2 resolves it.

**Options Considered:**

- **Fixed URL-safe alphabet only, no `--alphabet` flag.** Rejected — denying hex tokens from a tool called `ids` feels petty, and the use case is common ("give me a short random hex string for a filename suffix").
- **Arbitrary custom alphabet via `--alphabet <chars>` in v1.** Rejected for v1 — validation and edge cases (multi-byte, duplicates, very short alphabets) are enough work to justify landing in a focused v2.

## 5. Error (exit 125) on Flag / Type Mismatch

**Context:** Flags like `--length`, `--alphabet` only make sense with `--type nanoid`; `--format`, `--uppercase` only with `--type uuid4`/`uuid7`. What happens when a user combines them incorrectly?

**Decision:** Error at parse time, exit 125, with a specific message naming the offending flag and the types it applies to.

**Rationale:** The `clip --primary` precedent (silent ignore on non-Linux platforms) exists to support *portability* — the same script running on different platforms shouldn't fail. That reasoning does not apply here: the user is making both choices (`--type X` and `--flag-for-other-type`) in the same invocation, so there's no portability benefit to silent ignore. The likeliest cause of the combination is a mistake — a typo'd `--type`, a half-remembered flag, copy-pasted example. Erroring catches the mistake; silent ignore masks it. The message names the flag and the types it applies to, so the fix is obvious.

**Trade-offs Accepted:** Users who want to toggle `--type` in a shell loop while keeping other flags constant (`for t in uuid7 nanoid; do ids --type $t --length 12; done`) will hit errors on the UUID iteration. Acceptable — the loop author should branch on type, or split the invocations. Aliases with fixed flag values (`alias shortid='ids --length 12'`) that might break under `--type uuid7` are uncommon.

**Options Considered:**

- **Silent ignore** (matching `clip --primary`). Rejected — the portability argument that justified it there does not apply to same-invocation flag conflicts. Here, silent ignore just hides mistakes.
- **Warn on stderr, proceed.** Rejected — chatty in loops, and the warning is noise once the user confirms the behaviour. Also awkward when `--json` mode has clean stdout but noisy stderr.

## 6. Monotonic ULID Generation (Within the Same Millisecond)

**Context:** Generating 1000 ULIDs takes a few milliseconds on modern hardware, so most of a batch lands in the same ms bucket. Without monotonicity, IDs within a bucket sort by random portion — independent of generation order. What should `ids --type ulid --count 1000 > ids.txt` produce?

**Decision:** Increment the 80-bit random portion under lock when the timestamp doesn't change, rather than re-randomising. Overflow within a single ms is impossible (80 bits of headroom).

**Rationale:** "Time-ordered identifiers" is half the reason a user picks ULID over UUID v4. Breaking that guarantee within the most common usage pattern (batch generation) would be a genuine footgun. The fix is ~15 lines of code and one `Monitor.Enter`, unmeasurably fast in CLI single-process usage. Earlier drafts of this ADR assumed .NET 10's `Guid.CreateVersion7()` already did this internally; empirical testing during Task 6 showed it does not — the BCL re-randomises the sub-timestamp bits on each call. `Uuid7Generator` therefore carries the same application-level guard pattern as `UlidGenerator`, making both time-ordered types behave consistently. Downstream: `ids --type ulid --count 1000 | sort -c` succeeds, which becomes an integration test.

**Trade-offs Accepted:** Adds a per-instance lock and mutable state (`_lastMs`, `_lastRandom`). Contention is zero in the CLI path (single-process, single-thread loop), and correctness under multi-threaded library usage is a bonus rather than a goal. The random portion within a ms bucket is no longer "fully random" — it starts random and increments — but the cryptographic security properties of the leading bits are unchanged, and nothing in the ULID spec prohibits this (the canonical reference implementations all do it).

**Options Considered:**

- **No monotonicity — re-randomise every call.** Rejected once the "generation order != sort order for batch output" footgun was identified. The observable user-visible failure is too obvious.
- **Monotonicity only when an explicit `--monotonic` flag is passed.** Rejected — the default should be correct. Users who need the pure-random variant can use `--type uuid4`.

## 7. Rejection Sampling for NanoID Bias Elimination

**Context:** Mapping a random byte to an alphabet index via `byte % alphabetLength` introduces modulo bias when `alphabetLength` doesn't divide 256. For a 62-char `alphanum` alphabet, bytes 248–255 map to chars 0–5 more often than chars 6–61.

**Decision:** Use rejection sampling with a power-of-two mask. Pick `mask = nextPowerOfTwo(alphabetLength) - 1`. For each drawn byte, compute `b & mask`. If the result is `< alphabetLength`, emit the char at that index; otherwise reject and draw again.

**Rationale:** Modulo bias is a real cryptographic flaw — in a 20-char NanoID from a 62-char alphabet, some characters are slightly more likely than others. Rejection sampling produces uniformly distributed output. The overhead is negligible: for `alphanum` (62, mask 63) the rejection rate is ~3%, for `hex` (16, mask 15) it's 0%, for `url-safe` (64, mask 63) it's 0%. Reference implementations in JavaScript, Go, Rust all use this approach — matching them keeps our output statistically indistinguishable from theirs.

**Trade-offs Accepted:** ~5 extra lines of code vs the naive modulo version. Slightly more bytes consumed from the RNG than the output length would suggest (e.g. 20-char `alphanum` consumes ~21 bytes on average). RNG is cheap; no real cost.

**Options Considered:**

- **Naive `byte % alphabetLength`.** Rejected — introduces observable bias for non-power-of-two alphabets. Small but wrong; reference implementations do better.
- **Generate 32-bit ints and use `int % alphabetLength`.** The bias is smaller (bias zone is 1/2^32 instead of up to 8/256) but not zero. Rejected — rejection sampling costs essentially the same and produces exactly uniform output.

## 8. Per-Type `IIdGenerator` Interface, Not Static Switch

**Context:** Four ID types today; v2 adds at least three more (UUID v3/v5/v6, KSUID). How should type dispatch be structured?

**Decision:** Define `IIdGenerator` interface with a single `Generate(IdsOptions)` method. Implement per-type. Factory maps `IdType` enum → concrete generator.

**Rationale:** Matches the `IClipboardBackend` precedent. Benefits:

- Each generator is testable in isolation with hand-written fakes for `ISystemClock` / `ISecureRandom`. Adding a test file for a new type doesn't require understanding the others.
- Adding a new type in v2 means a new file and one line in the factory — no risk of breaking existing paths.
- State that belongs to one generator (the ULID monotonicity lock) stays scoped to that generator, not in a global switch.

**Trade-offs Accepted:** Slightly more scaffolding than a static `switch` in one file. Acceptable — the switch form does not scale past ~3 types before dispatch reads like a list of mini-handlers anyway.

**Options Considered:**

- **Static switch on `IdType` in a single `IdGenerator` class.** Rejected — every new type edits shared code, harder to test in isolation, and the ULID lock gets awkward to scope.
- **Strategy classes without a shared interface** (just static methods `Uuid4Gen.Generate`, `UlidGen.Generate`, called directly). Rejected — factory needs a common dispatch target; the interface costs nothing and gains testability.

## Decisions Explicitly Deferred

| Topic | Why deferred |
|---|---|
| Arbitrary custom NanoID alphabets (`--alphabet <literal>`) | Named presets cover ~all real usage; custom strings open validation edge cases best handled as a coherent v2 addition. |
| UUID v1/v3/v5/v6 | Niche compared to v4/v7; v3/v5 need a namespace+name input pair that changes the flag shape. Bundle as "UUID extras" v2. |
| CUID / CUID2 / XID / KSUID / Snowflake | Tier-2 ask at best; snowflake specifically needs machine-ID persistence, which is too much infrastructure for a single-binary tool. |
| Slug-style human-readable IDs | Rejected scope (per memory) — a separate tool if ever built. |
| Password / passphrase generation | Rejected — future `pass`/`pwgen` tool has separate requirements (word-lists, clipboard integration, length/charset semantics). |
| `--monotonic` flag to opt out of ULID monotonicity | Not needed in v1 — `--type uuid4` covers the "pure random" use case, and turning off ULID monotonicity would only exist for pedantic compatibility with non-monotonic implementations, which is not a real user need. |
