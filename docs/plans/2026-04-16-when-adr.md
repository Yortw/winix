# ADR: when Tool Design Decisions

**Date:** 2026-04-16
**Status:** Proposed
**Context:** Design for `when`, a date/time conversion and arithmetic tool for the Winix CLI suite.
**Related:** [when design doc](2026-04-16-when-design.md)

---

## 1. Multi-Line Default Output

**Context:** What should the default output look like when converting a timestamp?

**Decision:** Show all representations in a multi-line format: UTC, local, relative, and Unix epoch. With `--tz`, add the target timezone line.

**Rationale:** The primary use case is "I have this timestamp, what does it mean?" Showing all formats at once answers the question completely without requiring the user to re-run with different flags. It's only 4-5 lines — informative, not verbose.

**Trade-offs Accepted:** Slightly more output than a single-value tool. Mitigated by `--utc`, `--local`, and `--json` for pipe-friendly single-value output.

**Options Considered:**
- Single "opposite" conversion (epoch in → ISO out) — too clever, doesn't show relative time or other timezone
- Single-line default with --verbose — forces users to add a flag for the common case

---

## 2. Output to stdout (Not stderr)

**Context:** Other Winix tools output summaries to stderr to protect child process stdout. Should `when` follow this convention?

**Decision:** Output to stdout. No `--stdout` flag.

**Rationale:** `when` does not wrap a child process. Its output IS the primary output. Sending it to stderr would require `--stdout` for every piped use case (`when now --utc | ...`), which is bad ergonomics for a conversion tool. Errors still go to stderr.

**Trade-offs Accepted:** Inconsistency with other Winix tools' default output stream. Justified because the architectural reason for stderr (protecting child stdout) doesn't apply.

**Options Considered:**
- stderr default for consistency — would make piping awkward for the most common use case

---

## 3. Disable InvariantGlobalization for Timezone Support

**Context:** All Winix tools use `<InvariantGlobalization>true</InvariantGlobalization>` for smaller AOT binaries. This strips ICU, which is needed for timezone resolution.

**Decision:** Disable InvariantGlobalization for the `when` console app only. All other tools remain invariant.

**Rationale:** Timezone handling is core to `when`'s value proposition. Without ICU, IANA timezone IDs don't work on Windows, and cross-platform timezone ID resolution fails. Getting timezones right is more important than binary size for this specific tool.

**Trade-offs Accepted:** ~30MB larger AOT binary on Windows (bundled ICU). Linux/macOS unaffected (system ICU). The .NET global tool is unaffected (uses runtime ICU).

**Options Considered:**
- Ship a static IANA→Windows mapping table — maintenance burden, incomplete, fragile
- Platform-native IDs only — different timezone names on each platform, poor UX
- Stripped ICU with only timezone data — possible optimisation, deferred to later

---

## 4. Accept Both IANA and Windows Timezone IDs

**Context:** IANA IDs (`Asia/Tokyo`) are the standard on Linux/macOS. Windows uses its own IDs (`Tokyo Standard Time`). Which should `when` accept?

**Decision:** Accept both. With ICU enabled, .NET resolves both ID formats on all platforms transparently.

**Rationale:** Users shouldn't need to know which platform they're on. A developer who copies `Asia/Tokyo` from a config file or `Tokyo Standard Time` from a Windows error message should get the right result either way.

**Trade-offs Accepted:** None — this is free with ICU enabled.

---

## 5. Reject Ambiguous Date Formats

**Context:** Should `when` try to parse `06/12/2024` (is it June 12 or December 6?)?

**Decision:** Reject purely numeric slash/dash-separated dates. Accept only ISO 8601 and named-month formats for dates.

**Rationale:** Guessing DD/MM vs MM/DD would produce silently wrong results — the worst possible outcome for a date tool. Named-month formats (`Jun 18 2024`) are unambiguous. ISO format (`2024-06-18`) is unambiguous. The error message suggests ISO format.

**Trade-offs Accepted:** Users with `MM/DD/YYYY` habits must use ISO format. This is a feature, not a bug.

**Options Considered:**
- Locale-aware parsing — varies by system locale, silently wrong on different machines
- `--format` flag to specify input format — adds complexity, deferred to v2 if needed

---

## 6. Epoch Millisecond Auto-Detection by Digit Count

**Context:** How to distinguish epoch seconds from epoch milliseconds?

**Decision:** 10 digits or fewer = seconds. 11-13 digits = milliseconds (JavaScript convention). Decimals are always seconds.

**Rationale:** The 10/13 digit boundary is widely used (JavaScript `Date.now()` returns 13-digit milliseconds; Unix `time()` returns 10-digit seconds). The ambiguity window (10-digit values that could be either) doesn't exist in practice — 10-digit second timestamps cover 2001–2286, and 10-digit millisecond timestamps cover only 1970-01-12 (not useful).

**Trade-offs Accepted:** Values outside the typical ranges could be misinterpreted. Mitigated by the clear digit-count boundary and the rarity of edge cases.

---

## 7. ISO 8601 Duration: Days and Below Only

**Context:** ISO 8601 durations support years and months (`P1Y2M`), but these are calendar-dependent (a month is 28-31 days).

**Decision:** Support only days and below in v1 (`PnDTnHnMnS`). Reject `P1Y` and `P1M` with a clear error.

**Rationale:** `TimeSpan` cannot represent calendar-dependent durations. Adding calendar arithmetic would require NodaTime or significant custom code. Days and below are unambiguous and cover the practical use cases.

**Trade-offs Accepted:** Users who need month/year arithmetic must calculate manually. Revisit in v2 if demand emerges.

**Options Considered:**
- Assume 30-day months / 365-day years — silently wrong, violates correctness priority
- Pull in NodaTime — heavy dependency for one feature, AOT compatibility concerns

---

## 8. Diff Output Includes From/To Lines Only With --tz

**Context:** Should the diff output always show the resolved From and To timestamps?

**Decision:** The default diff output shows only Duration, ISO 8601, and Seconds. When `--tz` is specified, add From and To lines showing the timestamps in the requested timezone.

**Rationale:** The user already knows the inputs (they typed them). Echoing them back in the default case is noise. But with `--tz`, showing the timezone-converted inputs is useful context ("these are the times I compared, in Tokyo time").

**Trade-offs Accepted:** Without `--tz`, the user can't see how ambiguous inputs were resolved. Mitigated by the input parser's strict format requirements.

---

## Decisions Explicitly Deferred

| Topic | Why Deferred |
|-------|-------------|
| `--format` input format specifier | No demand yet. Strict auto-detection covers practical cases. |
| Month/year arithmetic (P1Y, P1M) | Calendar-dependent, needs NodaTime or significant custom code. |
| Stripped ICU bundle for smaller binary | Optimisation — measure actual size first, then decide. |
| `--epoch` output flag | `--json` provides `unix_seconds`. Single-value epoch output could be added if requested. |
| Named-day parsing (`next monday`) | Fuzzy and locale-dependent. Revisit in v2. |
| Duration arithmetic (add two durations) | Not a common use case. The tool converts timestamps, not durations. |
