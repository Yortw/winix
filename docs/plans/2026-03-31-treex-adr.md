# ADR: treex — Enhanced Directory Tree Viewer

**Date:** 2026-03-31
**Status:** Proposed
**Context:** Winix needs a tree viewer that fills the gap between DOS-era Windows `tree` and basic Linux `tree`, with colour, filtering, sizes, and modern terminal features.
**Related:** [treex Design](2026-03-31-treex-design.md), [files Design](2026-03-31-files-design.md), [CLI Conventions](2026-03-29-winix-cli-conventions.md)

---

## Decision 1: Separate TreeBuilder, not FileWalker reuse

**Context:** `Winix.FileWalk.FileWalker` yields a flat `IEnumerable<FileEntry>` stream. Tree rendering needs parent-child relationships, sibling position ("is this the last child?"), size rollups, and branch pruning.

**Decision:** `Winix.TreeX` has its own `TreeBuilder` that produces a `TreeNode` hierarchy. It reuses predicate helpers from `Winix.FileWalk` (GlobMatcher, SizeParser, DurationParser, ContentDetector) but not the `FileWalker` itself.

**Rationale:** A flat stream cannot express sibling relationships, which are essential for tree rendering (`├──` vs `└──`). Reconstructing a tree from a flat stream would be more complex and fragile than walking directly into a tree structure. The shared value is in the predicates, not the walking.

**Trade-offs Accepted:** Some directory enumeration logic is duplicated between FileWalker and TreeBuilder. Acceptable — the walking patterns are fundamentally different (stream vs tree), and the shared predicates capture the real overlap.

---

## Decision 2: Names-only default, size/date opt-in

**Context:** Tree viewers can show just names (clean), names + sizes (informative), or full details (cluttered). Users primarily reach for `tree` to see structure.

**Decision:** Default output shows only names. `--size` adds a right-aligned size column with human-readable units and directory rollups. `--date` adds a date column. Both composable.

**Rationale:** `tree` users expect names — familiar and clean. Sizes require `stat()` per file which adds I/O overhead. Structure is the primary value; detail is opt-in when you need it.

**Trade-offs Accepted:** Users who always want sizes need to type `--size`. Acceptable — aliases solve this for power users.

---

## Decision 3: Directories first, alphabetical sort default

**Context:** Linux `tree` sorts alphabetically with directories and files mixed. Many modern tools (VS Code, eza) put directories first.

**Decision:** Default sort is alphabetical with directories grouped first, then files. `--sort size` and `--sort modified` for alternatives. Directories always first regardless of sort mode.

**Rationale:** Directories-first matches developer expectations (VS Code, most file managers). Alphabetical within each group is predictable and scannable.

---

## Decision 4: Pruned tree for filtered output

**Context:** When filtering (e.g. `--ext cs`), should the tree show the full structure with highlights, or prune to only branches containing matches?

**Decision:** Prune to matching branches. Ancestor directories are kept to show path structure, but empty branches are removed.

**Rationale:** The pruned view answers "where are the matching files?" which is the question you're asking when filtering. A full tree with highlights would be too noisy for large directories.

---

## Decision 5: OSC 8 hyperlinks on by default

**Context:** Modern terminals (Windows Terminal, iTerm2, VTE) support clickable hyperlinks via OSC 8 escape sequences. Tree viewers show many file paths — making them clickable adds genuine utility.

**Decision:** File and directory names are OSC 8 hyperlinks using `file:///` URLs. On by default when colour is enabled (which implies a capable terminal). `--no-links` to disable.

**Rationale:** Click-to-open is a natural interaction for a tree viewer. Files open in the default editor, directories in the file manager. The implementation cost is low (URL wrapping) and the UX value is high.

**Trade-offs Accepted:** Terminals that support ANSI colour but not OSC 8 will show the escape sequences literally. Mitigated by: such terminals are increasingly rare, and `--no-links` is available.

---

## Decision 6: Dim tree connectors

**Context:** Tree connector characters (`├──`, `└──`, `│`) are structural scaffolding. They can be full brightness (like `tree`) or dimmed to let file names stand out.

**Decision:** Tree connectors are rendered with ANSI dim (`\x1b[2m`). Sizes and dates are also dim when shown.

**Rationale:** The names are the content — connectors are structure that guides the eye but shouldn't compete for attention. Dim connectors make the actual files and directories more scannable.

---

## Decision 7: Multiple root paths

**Context:** Should treex accept multiple root paths?

**Decision:** Yes. Multiple roots render as separate trees with a blank line between them. Matches `tree`'s behaviour.

**Rationale:** Trivial to implement (render each root sequentially). Useful for comparing directory structures. Consistent with `files` which also accepts multiple roots.

---

## Decisions Explicitly Deferred

| Topic | Why deferred |
|-------|-------------|
| `--follow` for symlinks | Not needed in v1. Symlink directories are shown but not followed. Add when needed. |
| `--text` / `--binary` filtering | Available in the shared predicates but not worth the per-file I/O cost for a visual tree. Use `files --text` instead. |
| Terminal width awareness | Output doesn't truncate long names in v1. Could add `--width` or auto-detect later. |
| Interactive mode | No scrolling/search in v1. Pipe to `less` for that. |
| `--du` mode | Show only directories with cumulative sizes (like `du`). Can be added as `--dirs-only --size --sort size`. |
| Colour by extension | Colour files by extension (.cs = green, .json = yellow, etc.). Too opinionated without configuration. |
