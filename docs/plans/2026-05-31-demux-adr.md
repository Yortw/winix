# demux — Architecture Decision Record

**Date:** 2026-05-31
**Status:** Accepted
**Context:** Adding `demux`, a stream-partition pipe filter, as a v0.4.0 tool. Companion to the
design doc: `2026-05-31-demux-design.md`. Class-4 "stream-plumbing" candidate from the tool
backlog; first of that family to be built.

---

## D1 — Build it at all (value vs. awk / pwsh `switch`)

- **Context:** Partition-by-pattern is partly served elsewhere — `awk` on Unix (a real pipe
  filter, cryptic), PowerShell `switch` on Windows.
- **Decision:** Build it.
- **Rationale:** The PowerShell `switch` is a *statement*, not a pipe filter — it cannot be piped
  into, cannot sit mid-pipeline, and its case bodies are script blocks, not command sinks. So there
  is a genuine *pipe-filter partition* gap on Windows (cmd has nothing either). On Unix, `awk` is
  the real competitor; the win there is readability + cross-platform consistency + agent-friendly
  `--json`/`--describe` composition.
- **Trade-offs accepted:** On Unix this is a consistency/ergonomics play (the `url` tier), not a
  hard gap-fill — and it costs far more to build than `url` did (process-orchestration, wargs-class).
  We accept a second-tier *value* at first-tier *cost* because the Windows pipe-filter gap is real
  and the tool serves the suite's agent-composition bet.
- **Options considered:** (a) Defer to v0.5.0 and ship glob expansion instead — rejected because the
  user opted to expand v0.4.0 scope and wanted a new tool. (b) Pivot to `merge`/`fanin` — held as a
  future Class-4 tool; `demux` chosen first. (c) Don't build, recommend awk — rejected on the
  Windows-gap + agent-ergonomics grounds.

## D2 — Name: `demux`

- **Context:** The intuitive names are collision-blocked.
- **Decision:** Ship as `demux`.
- **Rationale:** `demux` (demultiplexer) is the *precise* term for one-input → one-of-N-outputs by
  selector — exactly the default first-match behaviour. Collision-free on every shell; distinctive
  and searchable.
- **Trade-offs accepted:** Recognition gap — not guessable on sight by non-technical users.
  Mitigated by the tagline ("split one stream into many by pattern") and the suite's
  `--describe`/`llms.txt`/`docs/ai` discovery, which already cover guessability; the name is spent
  on findability, which nothing else covers.
- **Options considered:** `switch` (PowerShell keyword — blocked), `route` (`route.exe` — blocked),
  `split`/`sort` (coreutils — blocked), `branch` (git confusion), `bucket`/`sieve`/`sift`/`triage`
  (evocative but each rejected by the user as unclear or mis-connoting), `divert`/`dispatch` (more
  guessable but generic and poorly searchable).

## D3 — Targets: files AND commands, via self-contained 2-operand route flags

- **Context:** A route must bind a predicate to a target that is either a file or a command. The
  initial draft used adjacency pairing (`--case PATTERN` followed by a separate `--to`/`--exec`).
- **Decision:** Both target kinds are first-class. Each route is a single flag carrying both
  operands: `--to PATTERN FILE` and `--exec PATTERN CMD` (repeatable, order-independent); unmatched
  default via `--default-to FILE` / `--default-exec CMD`.
- **Rationale:** Self-contained route flags make a half-specified route **structurally
  impossible** — there is no "pending case" to leave dangling, so a whole class of user/agent
  errors cannot be *expressed* rather than merely being caught at runtime ("make illegal states
  unrepresentable", which serves the correctness-first value). Order-independent and more robust for
  agents to generate. Typed flags keep file-vs-command unambiguous.
- **Trade-offs accepted:** Route flags take two operands — mildly unusual for CLI flags — and the
  operand order (pattern, then target) must be learned. Multi-value repeatable options aren't native
  to ShellKit's flat model, so a thin custom arg-assembly pass is still required (bounded; precedent
  in `schedule`/`url`/`qr`).
- **Options considered:** **Adjacency pairing** (`--case P --to F`) — reads more naturally in
  English but is stateful and lets you express orphan/dangling flags that must then be validated at
  runtime; rejected on the correctness grounds above (this reverses the original draft, after review
  flagged the implicit pairing as the design's one real wart). **Every-target-is-a-command** (files
  via `tee`) — spawns a process per file route, walks back "files first-class". **Single-token
  `PATTERN:TARGET`** — delimiter-fragile (patterns/paths contain `:`, `/`, `=`).

## D4 — Match semantics: first-match default, `--all` broadcast

- **Context:** A record may match multiple cases.
- **Decision:** First-match wins by default; `--all` opts into broadcast (route to every match).
- **Rationale:** First-match matches the demux mental model and is predictable; `--all` covers the
  tee-with-filters case for one cheap flag.
- **Trade-offs accepted:** Two behaviours to test and document.
- **Options considered:** All-match only (diverges from the partition intuition); first-match only
  (loses the broadcast use-case).

## D5 — Unmatched records → stdout (passthrough)

- **Context:** Records matching no route, with no `--default-*`, must go somewhere.
- **Decision:** Pass them through `demux`'s own stdout; `--default-to`/`--default-exec` redirects
  them to a sink instead. The stderr summary always reports the unmatched count.
- **Rationale:** No silent data loss (the suite's diagnosability bias), maximally composable
  (remainder flows to the next pipe stage), and it places `demux` in the passthrough-stdout class
  whose `--json` envelope goes to stderr.
- **Trade-offs accepted:** Means matched records are *consumed* (not echoed to stdout); the model is
  siphon, not copy.
- **Options considered:** Drop + count on stderr (diagnosable but lossy); drop silently (rejected —
  silent loss); error unless `--default-*` (hostile to one-liners).

## D6 — `--exec` runs via the platform shell

- **Context:** Command targets like `gzip > rest.gz` need shell features.
- **Decision:** Spawn `--exec` commands via `sh -c` / `cmd /c`, passing the command as a single
  `ArgumentList` element.
- **Rationale:** Shell features are expected in the target; the command is author-supplied, so the
  trust model is the user's own `sh -c`. `ArgumentList` (shell + `-c`/`/c` + command) honours the
  no-arg-string-building convention.
- **Trade-offs accepted:** Shell interpretation = injection surface if a caller ever templates
  untrusted data into the command — documented as the author's responsibility.
- **Options considered:** Tokenize the command ourselves and exec directly — loses pipes/redirects
  and re-introduces quoting bugs the convention exists to avoid.

## D7 — Standalone tool, no speculative shared library

- **Context:** `demux` is the first of a potential Class-4 family.
- **Decision:** Build `Winix.Demux` as its own class library per suite architecture. Keep reusable
  orchestration (child sink, broken-pipe handling) as self-contained units, promotable to a shared
  library only when a second Class-4 tool exists.
- **Rationale:** Matches suite precedent (one tool per binary; `wargs`/`peep`/`nc` don't share an
  orchestration lib) and the YAGNI "no abstraction for one consumer" principle.
- **Trade-offs accepted:** A future second plumbing tool may require a small extraction refactor.
- **Options considered:** `plumb` multi-subcommand binary (commits unbuilt primitives to
  subcommands); extract a shared lib now (speculative).

## D8 — Dead-route-keeps-going on child broken-pipe

- **Context:** An `--exec` child can exit early, breaking its stdin pipe.
- **Decision:** Catch per-sink, mark the route dead, count undelivered, keep all other routes
  running; report in the summary. Exit `1` for the partial failure.
- **Rationale:** One failed child must not starve siblings or crash the router; the drop must be
  diagnosable, never silent. Diagnostic reporting is wrapped so it can't mask the data path.
- **Trade-offs accepted:** Records destined for a dead route are lost (counted, reported) rather
  than buffered or rerouted.
- **Options considered:** Abort the whole run on first broken pipe (too brittle for multi-route);
  reroute to default (surprising, hides the failure).

## D9 — `--exit-on-child-error` in v1

- **Context:** A child may run fine but exit non-zero.
- **Decision:** By default, a child's non-zero exit is reported but does not fail `demux`; the
  `--exit-on-child-error` flag makes it exit `2` (see D10). Included in v1.
- **Rationale:** `demux`'s job is routing, not the child's success — but CI users want a strict mode.
  Cheap to add; explicit opt-in matches the `wargs`/`retry` philosophy.
- **Trade-offs accepted:** One more flag and exit-code path to test.
- **Options considered:** Always fail on child non-zero (surprising for fire-and-forget routing);
  never (no CI strictness) — both rejected in favour of the opt-in.

## D10 — Distinct exit codes for delivery failure (`1`) vs watched-child failure (`2`)

- **Context:** Two non-success runtime conditions exist: a route dies mid-run and records are
  undelivered (data lost), and — under `--exit-on-child-error` — a child consumes everything but
  exits non-zero (no data lost). An initial draft collapsed both into exit `1`.
- **Decision:** Split them: `1` = partial delivery failure (data lost), `2` = watched child exited
  non-zero (delivery complete). If both occur in one run, **`1` wins** (data loss is more serious).
- **Rationale:** The two conditions call for *different operator responses* — `1` means "a sink died,
  check the child and whether you lost data"; `2` means "delivery was fine, your watched command
  failed its own check." Merging them forces the operator to re-derive the distinction from stderr,
  the diagnosability tax this suite avoids. Distinct low codes are consistent with the suite's
  structured-exit-code style and grep/diff precedent. The split also surfaces a co-occurrence
  precedence question the single-code design had silently ignored.
- **Trade-offs accepted:** One extra code and a precedence rule to document and test.
- **Options considered:** Single `1` for both (simpler, but discards the CI-relevant distinction and
  hides the co-occurrence question); pushing the child-error code higher (e.g. a 12x value) to keep
  `1` as the only "demux itself failed" code — rejected to keep common outcomes low and grep-like.
- **Note:** With shell-spawned `--exec`, "command not found" is the shell's `127` child exit
  (→ `2` under `--exit-on-child-error`), not demux's `126`; `126` is reserved for demux failing to
  open a `--to` file or launch the shell process.

## Decisions Explicitly Deferred

| Topic | Why deferred |
|---|---|
| NUL-delimited records (`-0`) | Line-oriented covers the dominant case; add on demand. |
| Shared Class-4 "plumbing" library | YAGNI until a second plumbing tool exists; units are promotable. |
| `merge`/`fanin` & other Class-4 primitives | Separate tools, separate brainstorms. |
| hcat-style capture/exit-on-match CI flags | Not obviously needed for a router; revisit if asked. |
| `--append` vs richer file modes | `--append` covers the log case; no rotation/size limits in v1. |
