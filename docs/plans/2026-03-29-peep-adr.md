# ADR: peep — Watch + File Watcher

**Date:** 2026-03-29
**Status:** Accepted
**Related design:** `2026-03-29-peep-design.md`

---

## 1. Combine watch + entr in a single tool

**Context:** Linux has `watch` (interval) and `entr` (file trigger) as separate tools. Windows has neither natively. viddy is a modern `watch` replacement but lacks file watching.

**Decision:** peep does both — interval polling, file-triggered runs, and combined mode (both simultaneously).

**Rationale:** The combined mode is the key differentiator. Nobody else offers "poll every N seconds AND re-run on file change" in a single tool. This is genuinely useful for development workflows (build on save, also poll periodically for external changes).

**Trade-offs Accepted:** More complex than a pure `watch` clone. Mitigated by clean separation (IntervalScheduler and FileWatcher are independent components).

**Options Considered:**
- *Interval only (like viddy):* rejected — misses the file-watch value proposition that makes peep unique.
- *File-watch only (like entr):* rejected — interval mode is table stakes and expected by `watch` users.

## 2. Alternate screen buffer for display

**Context:** Need to redisplay command output repeatedly without polluting terminal scroll history.

**Decision:** Use the terminal's alternate screen buffer (`\x1b[?1049h/l`).

**Rationale:** Clean display (no flicker), scroll history preserved, terminal restored on exit. This is what modern TUI tools (vim, less, htop, viddy) do. Supported by all modern terminals including Windows Terminal, iTerm2, and CLIo.

**Trade-offs Accepted:** Output disappears on exit. Mitigated by `--json-output` for scripts that need the last output, and `--once` for non-interactive use.

**Options Considered:**
- *Clear and redraw:* rejected — flickers, pollutes scroll history.
- *In-place overwrite:* rejected — complex when output length changes between runs.

## 3. Capture output rather than inherit terminal

**Context:** The child process's output needs to be displayed in a managed area of the alternate screen with a header.

**Decision:** Redirect child's stdout and stderr, capture into a buffer, then render to the alternate screen.

**Rationale:** Enables header line, output truncation, pause+scroll, and future diff highlighting. Colour output is preserved because ANSI sequences are captured and replayed.

**Trade-offs Accepted:** Commands that detect they're piped may disable colour. Workaround: `--color=always` or equivalent on the child command. This is the same trade-off viddy makes.

**Options Considered:**
- *Inherit terminal:* rejected — can't add header, can't truncate, can't pause+scroll.

## 4. Direct exec, no shell wrapper

**Context:** Linux `watch` passes commands through `sh -c`. This enables shell features (pipes, redirects) but adds shell injection risk and platform-specific behaviour.

**Decision:** Direct exec via `Process.Start`, consistent with timeit and the Winix convention. Shell features require explicit `bash -c "..."`.

**Rationale:** Safer, more predictable, cross-platform consistent. Most watch usage is simple commands that don't need a shell.

## 5. Pause mode with scroll

**Context:** `watch` has no way to read long output — it keeps refreshing. This is one of the most common complaints.

**Decision:** Spacebar pauses display while command keeps running. Arrow keys / PgUp / PgDn scroll through frozen output. Unpause shows latest result.

**Rationale:** Addresses a real usability gap. viddy's pause/scroll is one of its most-praised features. Cheap to implement — one boolean flag plus viewport offset over the captured buffer.

## 6. File watcher debounce default 300ms, configurable

**Context:** Build tools touch many files in rapid succession. Without debouncing, a single `dotnet build` could trigger dozens of re-runs.

**Decision:** Coalesce file change events — wait 300ms after first change for more events before triggering. Configurable via `--debounce`.

**Rationale:** 300ms is fast enough to feel responsive but long enough to batch typical build tool file operations. Configurable because some users may want snappier response (single file watch) or longer debounce (large builds).

## 7. JSON output on exit, not during

**Context:** peep runs in an alternate screen buffer during execution. JSON output needs to go somewhere useful.

**Decision:** JSON session summary written to stderr after peep exits. Optional `--json-output` includes last captured output text (ANSI stripped).

**Rationale:** JSON during execution would interfere with the alternate screen. On exit, the original terminal is restored and stderr is available. Scripts can capture the JSON for processing.

**Trade-offs Accepted:** No streaming JSON updates during execution. Acceptable for v1 — streaming would require a different architecture (sidecar log file or similar).

## Decisions Explicitly Deferred

| Topic | Why deferred |
|-------|-------------|
| Time machine / history navigation | Significant UI complexity — snapshot storage, backward/forward navigation. v2. |
| Diff highlighting (`-d`) | Requires line-level diff algorithm. v2. |
| Exit on regex match (`--exit-on-match`) | Natural v2 extension of auto-exit modes. |
| `.gitignore`-aware file watching | Needs glob parsing infrastructure, possibly ShellKit component. v2. |
| Text search (`/`) | Terminal-level search covers this. v2+. |
| SQLite persistence / lookback | v3+ — compelling but niche. |
| Shell wrapping | Consistent with Winix convention — direct exec only. |
