# ADR: Winix Less Pager

**Date:** 2026-04-11
**Status:** Proposed
**Related design:** `2026-04-11-less-design.md`

---

## 1. Windows-primary pager

**Context:** Windows has no native pager worth using. `more.com` is forward-only and destroys ANSI sequences. The `less` shipped with Git for Windows is an MSYS2 binary that only runs correctly inside a Git Bash context — it breaks in Windows Terminal, CMD, and PowerShell. Developers on Windows working with tools that produce paged output have no reliable, general-purpose option.

**Decision:** Build a native `less` pager as a cross-platform .NET binary, with Windows as the primary target and key value proposition.

**Rationale:** Fills a real gap on Windows where no adequate pager exists. Cross-platform build comes at low cost — the same .NET code runs on Linux and macOS — so there's no reason to artificially restrict the target even if the value proposition is weaker there.

**Trade-offs Accepted:** Limited incremental value on Linux and macOS where GNU less is universally available. The tool is a genuine improvement on Windows; on Unix it's an alternative that users are unlikely to prefer over GNU less.

**Options Considered:**
- *Ship a bundled MSYS2 less:* rejected — binary redistribution complexity, large dependency, still breaks outside MSYS2 contexts.
- *Use Windows Terminal's built-in scrollback:* rejected — not a pager; doesn't work for piped output, searching, or controlled pagination.

## 2. Modern defaults (FRX)

**Context:** GNU less was designed in 1983. Its defaults predate colour terminals and common short-output patterns. Git has shipped `LESS=FRX` as its default configuration for years precisely because the original defaults are considered harmful for modern use: long output gets paged even when it fits on one screen, raw ANSI sequences are not passed through, and the terminal is cleared on exit.

**Decision:** Default to quit-if-one-screen (`-F`), raw ANSI passthrough (`-R`), and no-clear-on-exit (`-X`). These are the effective defaults regardless of whether the user sets `LESS`.

**Rationale:** Matches the de facto convention established by Git. Short output (help text, brief listings) prints directly without requiring a keypress. ANSI colour always passes through correctly. The terminal state is preserved after paging so the user can scroll back.

**Trade-offs Accepted:** Differs from GNU less's out-of-the-box defaults, which could surprise users who rely on those defaults. The `-F` flag is silently accepted but already active, so explicitly passing it has no effect.

**Options Considered:**
- *Match GNU less defaults exactly:* rejected — the defaults are widely acknowledged as poor for modern terminals. Matching them would replicate known problems.
- *Make defaults configurable via a config file:* deferred — env var override (see decision 3) is sufficient for v1.

## 3. LESS env var as override, not additive

**Context:** Many tools (git, man, etc.) set or propagate the `LESS` environment variable. Users and tools rely on `LESS` to customise pager behaviour. The pager needs to interact predictably with existing `LESS` setups.

**Decision:** When `LESS` is non-empty, its value replaces the modern defaults entirely. When `LESS` is empty or unset, the modern defaults (FRX) apply.

**Rationale:** Declarative semantics — "here's exactly what I want" — are easier to reason about than additive merging. This also matches GNU less's own semantics: `LESS` is not a flag-appending mechanism, it defines the flag set. A user or tool that sets `LESS=iN` gets exactly that, which is what GNU less would give them.

**Trade-offs Accepted:** A user who has `LESS=iN` for case-insensitive search loses FRX defaults. This is the correct and consistent behaviour — they've expressed what they want. Users who want both must include FRX in their own `LESS` setting, which is the same discipline required with GNU less.

**Options Considered:**
- *Always apply FRX, merge with LESS:* rejected — additive merging creates unpredictable interactions when flags conflict.
- *Ignore LESS entirely, use hard-coded defaults:* rejected — breaks compatibility with tools that configure pager behaviour via LESS.

## 4. In-memory content loading

**Context:** The pager must handle output piped from other tools. Two approaches are feasible: stream content and buffer lazily as the user scrolls, or load all content into memory upfront before displaying.

**Decision:** Load all content fully into memory before displaying for v1.

**Rationale:** Simple to implement and reason about. Files of 100 MB (~1 million lines) are loaded comfortably within .NET's memory model. The typical use case — paging command output, man pages, log snippets — is well within this range. Streaming content while simultaneously handling keyboard input and ANSI rendering is a significantly more complex problem that is not warranted for v1.

**Trade-offs Accepted:** Cannot handle multi-gigabyte files without exhausting available memory. This is an acceptable v1 limitation: `less` is a reading tool, not a log analysis tool. Users with very large files can use specialised tools.

**Options Considered:**
- *Lazy streaming with ring buffer:* rejected for v1 — complex to implement correctly alongside search, ANSI parsing, and line wrapping. Worth revisiting if large-file support is requested.
- *Memory-mapped files:* rejected — adds complexity without benefit for the common case; also doesn't help for piped stdin, which can't be memory-mapped.

## 5. AnsiText in Winix.Less (not ShellKit)

**Context:** ANSI stripping and display-width calculation logic already exists in two places: `man`'s `BuiltInPager` and `peep`'s rendering layer. Both implement variants of the same logic — strip ANSI escape sequences to measure visible character width for line wrapping and display. Moving this to `Yort.ShellKit` would make it reusable across all tools.

**Decision:** Create a fresh `AnsiText` utility in `Winix.Less` for now rather than refactoring the shared library.

**Rationale:** Consolidating into ShellKit is a cross-cutting refactor that would require modifying and re-testing `man` and `peep` simultaneously. Blocking `less` on that refactor adds risk and delay for no user-visible benefit in v1. The less-specific implementation can be written clean and correct, and the consolidation can happen as a dedicated follow-up.

**Trade-offs Accepted:** Creates a third copy of similar ANSI logic across the codebase. This is intentional technical debt — acknowledged and accepted for now, with the expectation of consolidation once all three consumers are stable.

**Options Considered:**
- *Refactor into ShellKit before implementing less:* rejected — introduces coupling between unrelated deliverables and risk of regressing man and peep.
- *Copy from man's BuiltInPager directly:* considered but not preferred — better to write it clean for less and then consolidate all three at once.

## Decisions Explicitly Deferred

| Topic | Why Deferred |
|-------|-------------|
| Mouse wheel scrolling | Requires Win32 `ReadConsoleInput`; .NET `Console.ReadKey` does not capture mouse events |
| Streaming for large files | In-memory is fine for v1; streaming adds significant complexity |
| ShellKit ANSI utilities | Cross-cutting refactor that would touch man, peep, and less; better done as a dedicated step once all three are stable |
| Regex search patterns | Simple string search is sufficient for v1; regex adds escape handling and error reporting complexity |
| Search match highlighting | Requires per-character ANSI injection; adds visual complexity without being essential for v1 |
| Multiple file support | `:n`/`:p` navigation adds state management complexity; single-file paging covers all current use cases |
