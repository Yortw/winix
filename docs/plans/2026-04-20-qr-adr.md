# qr — Architecture Decision Record

**Date:** 2026-04-20
**Status:** Accepted
**Companion document:** [2026-04-20-qr-design.md](2026-04-20-qr-design.md)
**Context:** Designing `qr`, a Winix tool that generates QR codes offline. Filed against the v0.5.0 window (TBD); privacy-driven alternative to online QR generators for Wi-Fi credentials, crypto addresses, and internal URLs.

---

## D1. Flat default + helper subcommands (`qr <text>`, `qr wifi …`, `qr sms …`)

**Context.** Three shapes were viable: flat + flag-driven helpers (`qr --wifi …`), everything-as-subcommand (`qr text "…"`), or flat default with helper subcommands.

**Decision.** Flat default for text payloads; helper subcommands for Wi-Fi, SMS, mailto, geo, tel.

**Rationale.** The raw-text case (`qr "https://…"`) is 80%+ of expected usage and deserves zero ceremony — no subcommand word. Helpers benefit from isolated flag namespaces so that `--ssid` and `--password` only exist under `qr wifi`, not as globally-valid flags with positional collisions. The pattern is already in the suite via `url` (flat-ish `url encode` plus `url build` / `url join` / `url query`).

**Trade-offs accepted.** Subcommand dispatch is slightly more complex in `ArgParser` than a pure flag-driven shape. Offset: matches the existing `url` / `schedule` precedent so the pattern is already familiar.

**Options considered.**
- Flat + flag-driven (`qr --wifi --ssid X`) — rejected; flag surface grows with helper count, validation of "required-flag clusters per helper" is awkward.
- Everything-as-subcommand (`qr text "…"`) — rejected; extra word for the dominant case.

## D2. Output format: unicode on TTY, SVG when piped

**Context.** `qr "text"` with no `--format` needs a default. Always-unicode breaks pipe redirects to image files. Always-PNG produces binary on a TTY. Auto-detection by TTY is the remaining option; the question is what to emit when redirected.

**Decision.** `--format auto` (default): `unicode` on TTY, `svg` when stdout is redirected. `--format {unicode,ascii,svg,png}` overrides. PNG to a TTY is refused without `--force-binary`.

**Rationale.** SVG is text, so it doesn't corrupt terminals when a user forgets to redirect a pipe; it's scannable by any phone camera via any image viewer; it scales without loss. PNG stays one flag away for users who specifically need raster. Unicode on TTY matches the "paste this in a chat message or scan it off my screen" workflow. PNG-to-TTY refusal follows `squeeze` precedent — binary-to-terminal is almost always a bug.

**Trade-offs accepted.** A user running `qr "x" | cat` sees SVG XML, which may surprise them until they read `--help`. Mitigation: the man page and README document the auto-behaviour; `--format unicode` is the explicit opt-in for "always terminal art".

**Options considered.**
- Always unicode — rejected; pipe-to-file for image workflows would need an extra flag every time.
- Unicode on TTY, PNG when piped — rejected; PNG is binary, worse pipe corruption blast radius when someone forgets to redirect.

## D3. Ship five URI-style helper subcommands in v1 (`wifi`, `sms`, `mailto`, `geo`, `tel`). Defer vCard and iCal.

**Context.** The original tool sketch named seven helpers: Wi-Fi, vCard, iCal, SMS, mailto (plus geo and tel as obvious additions). The cost-per-helper varies.

**Decision.** Ship `wifi`, `sms`, `mailto`, `geo`, `tel` in v1. Defer `vcard` and `ical` to v2.

**Rationale.** The five shipped helpers share a uniform structural pattern — they all emit a single-line `scheme:opaque?params` URI with RFC 3986 percent-encoding. They share one test harness, one escape routine, one validation pass. `vcard` and `ical` are a different beast: multi-line records, their own line-continuation rules, and (for iCal) RFC 5545 timestamp + timezone semantics. Adding them in the same batch would double the spec size with distinct concerns. They deserve their own brainstorm.

**Trade-offs accepted.** "Full contact sharing" and "calendar event QR" are not covered in v1. Most users who want those reach for a phone contact card or a calendar link anyway; the privacy-angle killer feature is Wi-Fi, which does ship.

**Options considered.**
- All seven in v1 — rejected; vCard/iCal add disproportionate complexity.
- Wi-Fi only in v1 — rejected; the other four URI-style helpers are nearly free because they share the encoder pattern.

## D4. Extract `Winix.QrCode` shared library from day one

**Context.** The only known second consumer is a hypothetical `http --qr` feature (print the bind URL as a QR when `http` serves a LAN directory). `http` is deferred. YAGNI would say: inline in `Winix.Qr`, extract later.

**Decision.** Extract `Winix.QrCode` as a shared library from the initial implementation. Contains `QrEncoder`, `QrMatrix`, `EccLevel`, and the four renderers.

**Rationale.** User's explicit preference, overriding the YAGNI default. The upside is that the API boundary between "encode/render" and "payload-build/CLI" is set before a second caller starts making assumptions. The downside (one extra csproj + one extra test csproj) is small and consistent with `Winix.FileWalk` / `Winix.Codec` precedent where shared libs also started with one consumer.

**Trade-offs accepted.** Slightly larger initial project graph and one more `<ProjectReference>` edge. No NuGet-publishing commitment — shared libs so far are internal only, matching `Winix.FileWalk` and `Winix.Codec`.

**Options considered.**
- Inline in `Winix.Qr`, extract when `http` lands — initial recommendation; overruled by user preference for upfront extraction.

## D5. Default error-correction level M

**Context.** QR error-correction has four levels (L=7%, M=15%, Q=25%, H=30%). Higher levels produce larger codes but tolerate more damage.

**Decision.** Default `-e m`. `-e {l,m,q,h}` overrides.

**Rationale.** Matches `qrencode`, QRCoder's library default, and most "which ECC should I use" blog posts. A louder default (Q or H) would surprise users benchmarking output size against familiar generators and produce visibly larger terminal renderings. A quieter default (L) prioritises size over robustness, which is backwards for a tool whose most-cited use case (Wi-Fi stickers on a fridge) benefits from robustness.

**Trade-offs accepted.** Users printing Wi-Fi stickers for outdoor signage or camera-unfriendly conditions need to remember `-e h`. README and man page document the trade-off.

**Options considered.**
- Default L — rejected; too fragile for the physical-media use case the tool advocates.
- Default Q — rejected; larger-than-expected codes vs familiar generators.

## D6. Hand-roll helper payloads instead of using `QRCoder.PayloadGenerator`

**Context.** QRCoder ships `PayloadGenerator.WiFi`, `.Mail`, `.SMS`, `.Geolocation`, `.PhoneNumber` helper classes. Using them would save ~200 lines.

**Decision.** Hand-roll each helper in `src/Winix.Qr/Helpers/`. Do not take a dependency on `QRCoder.PayloadGenerator.*`.

**Rationale.** We need test coverage on edge cases (SSID/password with semicolons/colons/backslashes; mailto body with `&`/`=`; geo out-of-range lat/lon) regardless of whether the implementation comes from QRCoder or from our own code. Hand-rolling gives us:

1. Control over escaping details we've already committed to in the spec (e.g., Wi-Fi per-char backslash escaping matching the de-facto ZXing convention, which QRCoder's implementation may or may not match byte-exact).
2. Independence from QRCoder's versioning — we can upgrade QRCoder for encode-engine fixes without re-validating helper payload output.
3. Simpler trim/AOT analysis surface — fewer types crossing the boundary.

The helpers are each 20–40 lines. Not a significant write.

**Trade-offs accepted.** We own the maintenance of five small formatters. Upside: identical test surface we'd need anyway; downside: if a Wi-Fi URI spec clarification ships, we update our code instead of bumping a dep. Acceptable.

**Options considered.**
- Use `QRCoder.PayloadGenerator.*` directly — rejected; opaque escaping, version-coupled, still needs our own tests to validate output.
- Use QRCoder's helpers as an internal sanity check (shadow implementation) — rejected; double-maintenance cost.

## D7. `QRCoder.PngByteQRCode` for PNG (no System.Drawing dependency)

**Context.** QRCoder has three PNG renderers: `PngByteQRCode` (built-in, no deps), `QRCode` (System.Drawing.Common — Windows-only-ish on modern .NET), and external packages like `QRCoder.SystemDrawing`. AOT-cross-platform support is the deciding factor.

**Decision.** Use `QRCoder.PngByteQRCode` exclusively for PNG output.

**Rationale.** `System.Drawing.Common` is flagged Windows-only on .NET 6+ (via `[SupportedOSPlatform("windows")]`), which breaks our Linux/macOS publish targets. `PngByteQRCode` assembles PNG chunks manually with no image-library dependency — AOT-clean, cross-platform, trim-clean. It's the shipped recommendation in QRCoder's own AOT guidance.

**Trade-offs accepted.** `PngByteQRCode` gives us monochrome PNG only, no colour control. We don't need colour for v1 (out of scope), so no trade-off lands.

**Options considered.**
- `QRCode` + `System.Drawing.Common` — rejected; not cross-platform on modern .NET.
- Hand-roll PNG (emit IHDR/IDAT/IEND chunks ourselves) — rejected; `PngByteQRCode` already does this correctly, re-implementing adds risk.

## Decisions Explicitly Deferred

| Topic | Why deferred |
|---|---|
| Image-decode (QR image → text) | Requires an image-decode library (ZXing.Net candidate); AOT validation not yet done. v2 feature. |
| vCard helper subcommand | Multi-line records with escaping rules distinct from URI-style helpers. Own brainstorm. |
| iCal helper subcommand | RFC 5545 timestamp/timezone semantics warrant dedicated design. Own brainstorm. |
| Custom colours (`--fg`, `--bg`) | Cosmetic; unicode and PNG output both work monochrome for the privacy use case. Add when asked. |
| Embedded-logo composition | Cosmetic; increases ECC-level requirement, adds image-blend complexity. |
| Batch mode (multi-payload input) | Shell loops cover it. |
| Micro-QR, Structured Append, Kanji mode | QR Model 1 covers the target use cases. |
| `--copy` flag | Compose with `clip`: `qr "…" --format svg \| clip`. Keeps `qr` focused. |
| NuGet-publish `Winix.QrCode` as public package | Matches `Winix.FileWalk` / `Winix.Codec` precedent — shared libs stay internal until external demand appears. |
