# protect/unprotect Format Hardening — ADR

**Date:** 2026-04-29
**Status:** Proposed
**Context summary:** Round-1 fresh-eyes review of `protect`/`unprotect` (still on `release/v0.4.0`, not yet shipped) found four classes of defect in the WPRT file format and library implementation: cross-file chunk substitution on AEAD (C1), DPAPI ignores AAD entirely allowing intra-file reorder (C2), platform-gated tests use the forbidden early-return pattern (C3), and `AeadBackend` silently overwrites a stored key whose length is wrong (C4). Plus eight Important findings around overwrite safety, partial-output cleanup, exit codes, durability, and key lifetime. Since the tool is unshipped, we can modify the file format in place rather than version-bump.
**Related plan:** `docs/plans/2026-04-29-protect-format-hardening-plan.md`

---

## Decision 1: Add a 16-byte random `FileId` to the WPRT header (in-place format change)

**Context.** The current AAD on the AEAD path is `headerBytes(6) + chunkIndex(8) + isFinal(1)`. The 6 header bytes are `'WPRT' + 0x01 + <marker>` — fixed for any given backend. Two `.prot` files produced by the same backend share identical AAD inputs at every chunk index, so a chunk lifted from file B can be substituted at the same offset in file A and AES-GCM tag verification succeeds. The DPAPI backend ignores AAD entirely, so the same substitution plus intra-file chunk reorder both succeed silently on Windows.

**Decision.** Extend the WPRT header from 6 to 22 bytes by appending a 16-byte random `FileId` generated by the writer per file. Bind `FileId` into AAD on AEAD backends, and into the DPAPI plaintext envelope (since DPAPI authenticates only what's inside the protected blob).

**Rationale.**
- A random per-file value is the minimum addition that makes "this chunk belongs to *this* file at *this* index" something the integrity layer can verify.
- 16 bytes is overkill for collision but fits the security-tool convention (matches GUIDs, AES block size, common nonce-derivation patterns) and the cost is negligible.
- Putting it in the header (not derived from the key) means it's free to read on decrypt without any keystore interaction.

**Trade-offs accepted.**
- Header grows from 6 to 22 bytes per file — trivial.
- The format breaks backward-compat with any pre-shipped builds, but per the `release/v0.4.0` status none exist outside the dev tree.
- We deliberately keep the version byte at `0x01` rather than bumping to `0x02`; the format is unshipped and bumping the version with no rollback story would be cargo-cult.

**Options considered.**
1. **Bump version to `0x02`** — rejected: nothing to be backward-compatible with; the bump would be ceremony, not function.
2. **Derive the per-file binding from the key** (e.g. HKDF over key + file path) — rejected: brittle if the file moves, and conflates "where the file lives" with "what the file is."
3. **Document the integrity gap and ship as-is** — rejected: tool's marketing promise is encrypt-at-rest with tamper detection; integrity-on-paper-only doesn't hold up to readme scrutiny.
4. **AAD-only fix on AEAD** (don't touch DPAPI) — rejected: leaves DPAPI as the weakest link of an otherwise uniform format. We commit to a uniform integrity model across backends.

## Decision 2: DPAPI binds `FileId` and `chunkIndex` inside the plaintext envelope, not via AAD

**Context.** `ProtectedData.Protect` doesn't accept an AAD parameter at all — DPAPI authenticates only what's inside the blob. Today the DPAPI backend takes an `AadContext` parameter and silently ignores it.

**Decision.** Change the DPAPI plaintext envelope from `[isFinal(1) | plaintext]` to `[isFinal(1) | FileId(16) | chunkIndex(8) | plaintext]`. On decrypt, verify that the embedded `FileId` matches the header's `FileId` and that `chunkIndex` matches the position the reader expected. Surface mismatches as `CryptographicException` (so the catch tree in `Cli.Run` keeps working).

**Rationale.**
- DPAPI doesn't expose AAD, but it does authenticate everything inside the blob. Putting the binding *inside* the framed plaintext gets us the same integrity guarantee.
- Verifying both FileId AND chunkIndex on decrypt closes both the cross-file substitution (FileId mismatch) and the intra-file reorder (chunkIndex mismatch) paths.

**Trade-offs accepted.**
- Each DPAPI chunk grows by 24 bytes plaintext-pre-Protect. Negligible overhead.
- The DPAPI envelope and the AEAD AAD now have different shapes (binding is in different places). That's inherent to the underlying primitives — DPAPI has no AAD slot.

**Options considered.**
1. **Pre-MAC the AAD with HMAC under a separate key** — rejected: introduces a second key, defeats the "zero key management" promise.
2. **Wrap DPAPI in AES-GCM ourselves** (AEAD over a DPAPI-protected key) — rejected: re-architects the backend for marginal gain over envelope-binding.

## Decision 3: Centralize header serialization in `Header.cs`

**Context.** The 6-byte header literal `[(byte)'W', (byte)'P', (byte)'R', (byte)'T', 0x01, (byte)backend.Marker]` is hand-built at five call sites: `Cli.cs:125,139,185`, `InPlaceExecutor.cs:34,71`, and the equivalent in `RoundTripVerifier.cs:30`. The `Header.Write` method exists but is bypassed. This is a drift hazard — if the format changes (which it now will), some sites will get updated and others won't, and the failure mode is a `CryptographicException` blamed on user/machine mismatch.

**Decision.** Add `Header.SerializeForAad(PlatformMarker, ReadOnlySpan<byte> fileId) -> byte[]` for AAD construction, and `Header.WriteHeader(Stream, PlatformMarker, ReadOnlySpan<byte> fileId)` for writing. `Header.Read` returns a `ReadResult` that includes the FileId bytes. All five hand-built sites change to call these helpers.

**Rationale.**
- One producer of header bytes means the format can only drift in one place.
- The AAD is conceptually `headerBytes + chunkIndex + isFinal` — keeping `headerBytes` derived from `Header` rather than reconstructed by callers ties the AAD to the actual on-wire bytes.

**Trade-offs accepted.**
- Adds one indirection layer; small.

**Options considered.**
1. **Make `Header.ReadResult` a non-record struct with a private constructor** to enforce that only `Header.Read` produces it — deferred. Useful but not load-bearing for the format change.

## Decision 4: Fail closed on wrong-size stored key (no self-heal)

**Context.** `AeadBackend.GetOrCreateKey` returns the cached key only if it exists *and* is `KeySize` (32) bytes. Otherwise — *including the case where it exists but is the wrong size* — it generates a fresh random key and writes it back to the keystore, overwriting whatever was there. If a future format change extends the key, or if the keystore entry gets corrupted, all existing `.prot` files become permanently undecryptable with no warning.

**Decision.** When the existing key is non-null but wrong size, throw `InvalidOperationException` with a message identifying the namespace and key name and instructing the user to remove the stored entry manually. Do not auto-regenerate.

**Rationale.**
- Silent overwrite when the consequence is total data loss is the wrong default for any value the user might consider precious.
- Manual remediation (delete keychain/libsecret entry) is well-defined and easy to document.

**Trade-offs accepted.**
- Slightly worse UX in the (hopefully theoretical) case where the keystore entry is corrupt — user sees an error instead of silent self-heal.

**Options considered.**
1. **Auto-rotate with a "previous key" slot** — rejected: extra storage, extra failure modes, not warranted given the user-tied threat model.

## Decision 5: Default behaviour refuses to overwrite an existing destination; opt in with `--force`

**Context.** `Cli.RunProtect` and `Cli.RunUnprotect` open the destination with `FileMode.Create`, which truncates an existing file silently. `unprotect secrets.json.prot` clobbers any existing `secrets.json` with no warning.

**Decision.** Default to `FileMode.CreateNew` (errors if dest exists). Add `--force` / `-f` flag that opts in to overwrite. Emit a clear error message naming the destination path.

**Rationale.**
- "Don't silently overwrite the user's data" is the safe default for a CLI tool.
- `--force` is the conventional opt-in (cp, mv, rm all use this pattern).

**Trade-offs accepted.**
- Small UX friction on re-runs (`protect file.json` second time errors instead of silently overwriting `file.json.prot`). Acceptable; user adds `-f`.

**Options considered.**
1. **Smart default: overwrite when output is auto-derived (`.prot` suffix), require `--force` only on explicit `-o`** — rejected: subtle behavioural rule users have to remember.

## Decision 6: Adopt `winix-protect/keys` as the AEAD secret-store namespace (post-divergence)

**Discovered during Task 15 Step 3 (WSL smoke) on 2026-04-30. Recorded here per `feedback_plan_to_code_divergence_must_be_recorded.md` as plan-to-code divergence.**

**Context.** `AeadKeychainBackend` and `AeadLibsecretBackend` both use the literal `"winix-protect"` as the secret-store namespace. On 2026-04-22 (commit `6340999`), `LinuxLibsecretStore` was tightened to require `<tool>/<sub...>` format via `LinuxNamespace.ExtractTool`. `"winix-protect"` lacks a slash, so the libsecret backend has been unconditionally broken on first key access since that date. End-to-end Linux smoke caught it; helper-level unit tests didn't, because they covered the helper's edge cases without asserting that backend constants satisfy the contract.

**Decision.** Both AEAD backends now share `SecretLayout.KeyNamespace = "winix-protect/keys"` — the `winix-protect` tool prefix, with `keys` as the sub-namespace. A new `AeadBackendNamespaceContractTests` asserts the constant satisfies `LinuxNamespace.ExtractTool` so any future rename trips a test.

**Rationale.**
- `<tool>/<sub>` is the existing convention (e.g. envvault uses `envvault/<user-namespace>`); adopting it for protect aligns the suite.
- A single shared constant in `SecretLayout.cs` prevents the two backends drifting apart on the namespace — drift on key location would silently lose access to existing data.

**Trade-offs accepted.**
- Both backends now share a layout file (small added indirection). Worth it — the previous duplicated literal is exactly the kind of thing that drifts.
- Constant-validation test is unit-level, not integration-level. A stronger guard would require running gnome-keyring-daemon in CI, which is non-trivial; deferred to the existing F10 CI follow-up.

**Options considered.**
1. **Loosen the `LinuxLibsecretStore` namespace contract to accept bare tool names** — rejected: deliberately tightened on 2026-04-22 to make sub-namespacing mandatory; reverting would weaken envvault's invariants.
2. **Migration shim that reads from old `"winix-protect"` namespace and re-stores under new** — rejected: nobody has a key under the old name (Linux has been broken since the contract change; macOS protect is unreleased). Migration logic would be dead code from day one.
3. **Per-backend namespace constants (Keychain uses `"winix-protect"`, Libsecret uses `"winix-protect/keys"`)** — rejected: makes the backends behaviourally divergent on a non-platform-specific concern. Single shared constant is clearer.
