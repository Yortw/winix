# hcat — Architecture Decision Record

**Date:** 2026-05-30
**Status:** Accepted
**Context:** Designing `hcat` (`Winix.HCat`), the "netcat for HTTP" tool — last in the v0.4.0 batch. Decisions below were made during the brainstorm; the full design is in [`2026-05-30-hcat-design.md`](2026-05-30-hcat-design.md).

---

## D1 — Server framework: Kestrel via `WebApplication`, at the middleware/`HttpContext` level

**Context.** hcat needs a cross-platform HTTP server that AOT-publishes cleanly, gives full control of request/response streams, and supports static-file serving + TLS without reinvention.

**Decision.** Host ASP.NET Core via `WebApplication.CreateSlimBuilder` + Kestrel, and handle requests with **catch-all terminal middleware on raw `HttpContext`** — not the Minimal API endpoint-binding layer, not MVC.

**Rationale.** A spike (see D8) proved `CreateSlimBuilder` + Kestrel + `UseStaticFiles` + `UseDirectoryBrowser` + source-gen JSON AOT-publishes with zero warnings and runs. Operating at the middleware level sidesteps Minimal-API endpoint/binding limitations entirely (we read `HttpContext` directly), and the middleware pipeline is ASP.NET Core's most extensible layer — every deferred v2 feature (auth, allowlist, fault injection, pipe routing, reverse proxy) is just more middleware. It is the least-constraining option.

**Trade-offs accepted.** ~10 MB binary (the ASP.NET Core tax) vs ~1–2 MB for typical winix tools. AOT discipline required: source-gen JSON, no reflection-heavy add-ons.

**Options considered.** *MVC/Controllers* — rejected: not AOT-supported, unwanted conventions. *Raw `KestrelServer`* — rejected: loses the static-file/HTTPS middleware we want for free. *`System.Net.HttpListener` (hand-rolled)* — rejected: weak cross-platform TLS, no middleware ecosystem, more code; was only the fallback had AOT failed.

## D2 — Bind safety: localhost-only by default everywhere; LAN exposure is always explicit

**Context.** A bare, zero-friction `hcat` that serves the cwd is exactly when a user has *not* thought about network exposure. The category norm (`python -m http.server`, `miniserve`) binds all interfaces by default.

**Decision.** Default bind is `127.0.0.1` for every invocation (bare and explicit). LAN exposure requires an explicit `--lan` (`0.0.0.0`) or a non-loopback `--host`. The QR/LAN banner appears only when actually exposed.

**Rationale.** "Never silently expose the network" fits the suite's security posture. The asymmetry is the safety mechanism: only the dangerous choice must be typed; the safe choice is the default. `hcat serve ./preview` for a local look still works with no friction; sharing is a deliberate `--lan`.

**Trade-offs accepted.** Diverges from the category's expose-by-default norm; the headline LAN-transfer feature needs one flag (`--lan`) in the common case — judged honest, not a burden.

**Options considered.** *Default `0.0.0.0` + loud banner* — rejected: silently LAN-exposes the cwd on bare `hcat`. *Intent-graded (bare=local, `serve`=LAN)* — rejected as more complex than needed. *Require an explicit bind choice on `serve` (error otherwise)* — rejected: adds friction to the safe-and-common local case without extra safety over "localhost-unless-`--lan`."

## D3 — Command surface: subcommand tool with a default `serve cwd`, verb-or-error disambiguation

**Context.** Capabilities cluster into serve/inspect/pipe; bare `hcat` should be useful.

**Decision.** Dispatch on `positional[0]` (`serve`/`inspect`/`pipe`); bare `hcat` (no subcommand) serves the cwd. An unknown first positional is a usage error with a hint — a specific dir requires `hcat serve <dir>`, never `hcat <dir>`.

**Rationale.** Matches the suite's subcommand precedent while honouring strong "serve here" muscle memory (`python -m http.server`). The verb-or-error rule keeps the parser unambiguous and avoids a directory named `inspect` silently dispatching to a mode.

**Trade-offs accepted.** `hcat ./public` does not serve `./public` (must use `serve`); a dir literally named like a verb needs the explicit `serve` verb.

**Options considered.** *Pure subcommands, bare = usage* (A) — rejected: loses the strong serve-cwd convenience. *`hcat <non-verb>` serves it* (Option 2) — rejected: reintroduces verb-vs-path ambiguity and silent typo-misdispatch.

## D4 — Recording format: JSONL only

**Context.** `inspect` records incoming requests; the live `--json` output already produces one object per request.

**Decision.** JSONL is the only recording format — `hcat inspect --json > capture.jsonl`. The live stream *is* the recording; the echoed response body and the captured line share one request-record schema.

**Rationale.** Suite-native (`grep`/`jq`/streamable), zero extra format machinery, serves both dev "watch requests" and CI "capture and assert."

**Trade-offs accepted.** No direct import/replay into browser devtools/Postman.

**Options considered.** *HAR* — deferred to v2: real interop but verbose and off-label for server-side capture. *`.http`* — deferred: request-only and replay needs a client hcat lacks in v1.

## D5 — pipe: single-command CGI primitive

**Context.** "Route requests to a CLI command" is the headline feature.

**Decision.** `hcat pipe -- <cmd>` runs one command per request with CGI-style mapping (body→stdin, stdout→response, metadata→env vars, exit `0`→200/non-zero→500, stderr→console).

**Rationale.** The true "netcat for HTTP" shape — a sharp single primitive, genuinely useful (expose a script/filter over HTTP), minimal surface. CGI conventions are familiar.

**Trade-offs accepted.** No per-path routing or command-set status in v1.

**Options considered.** *Path-routing table in v1* — deferred to v2: a "framework" feature that pulls against the single-primitive positioning.

## D6 — `inspect` echoes the request by default

**Context.** `inspect` serves both as a webhook sink (just needs 2xx) and a request inspector (httpbin-style reflection).

**Decision.** Default response is `200` with a JSON body echoing the request; `--status <code>` overrides the status.

**Rationale.** One default that satisfies both roles — the sender gets a 2xx *and* sees what it sent.

**Trade-offs accepted.** Canned-body / mock-response behaviour is not in v1 (only the status is configurable).

**Options considered.** *Bare 2xx / 204* — rejected as less useful than echo. *Fully configurable canned response* — deferred to v2.

## D7 — `serve --upload` is opt-in; HTTPS uses an in-memory self-signed cert

**Context.** Uploading-to-disk and TLS both add surface.

**Decision.** The POST upload receiver is off unless `--upload`. Uploads default to a dedicated `./uploads` directory, created if missing and **excluded from serving** (secure default — uploads aren't downloadable or listed). `--upload-dir <path>` relocates it. The governing invariant is **"uploads are downloadable iff they physically sit inside the served tree"**, so `--upload-dir .` (the served root) is the explicit, banner-warned escape hatch for a downloadable shared drop. Path-safety always applies: base-name only, reject `..`/absolute, resolve under the upload root, collision-suffix never overwrite. `--https` generates a self-signed cert fresh in-memory per run (no files, no trust-store changes).

**Rationale.** Read-only serving is the safe default; writing to disk is a sharper edge that should be deliberate and secure-by-default (uploaded files shouldn't silently become downloadable to everyone on the LAN). Expressing exposure as the single invariant above keeps the model contradiction-free and makes the in-root case an explicit, visible choice. An in-memory cert needs no key management and suits dev/LAN (the client accepts the warning).

**Trade-offs accepted.** Uploads need a flag and the secure default requires hcat to actively exclude the upload dir from serving; the self-signed cert triggers a client trust warning (acceptable for dev/LAN).

**Options considered.** *Uploads into the served dir by default* — rejected: uploaded files become instantly downloadable/listed to everyone reachable. *Uploads on by default* — rejected (unexpected write surface). *`--serve-uploads` flag to serve a named upload subfolder in place* — deferred to v2 (the `--upload-dir .` escape hatch already covers "downloadable uploads"). *Persisted/trusted cert* — deferred (`--cert <file>` BYO-cert is a v2 item Daisy flagged).

## D8 — AOT feasibility validated by spike before committing the architecture

**Context.** Project memory flagged "AOT trim warnings on Kestrel may bite" as a possible go/no-go.

**Decision.** Ran a minimal spike (`Microsoft.NET.Sdk.Web`, `CreateSlimBuilder` + Kestrel + `UseStaticFiles` + `UseDirectoryBrowser` + source-gen JSON) published `-r win-x64` with `PublishAot`. Result: **zero AOT/trim warnings, 9.8 MB binary, runs** (`curl /echo` returned source-gen JSON). Architecture is GO.

**Rationale.** Verify-before-asserting on the single most architecture-shaping risk; mirrors the blob-tool "spike before brainstorm" precedent.

**Trade-offs accepted.** None — the spike retired the risk.

## Decisions Explicitly Deferred

| Topic | Why deferred |
|---|---|
| pipe path-routing tables | Framework creep; single-command primitive ships first. |
| Finer pipe status control (command-set status) | v1 keeps `0`→200/nonzero→500; revisit with real use. |
| `--cert <file>` (bring-your-own cert) | In-memory self-signed covers dev/LAN; BYO-cert is a clean v2 add. |
| `--serve-uploads` (serve a named upload subfolder in place) | `--upload-dir .` already covers downloadable uploads; named-visible-subfolder is a niche refinement. |
| HAR / `.http` export | JSONL covers v1; interop/replay are v2 if asked. |
| Latency/fault injection, basic auth, IP allowlist | All v2 middleware; not core to the v1 pitch. |
| Reverse proxy (YARP) | v2 — verify YARP AOT support first. |
| mDNS advertise, request replay, markdown render, Windows firewall rule | Out of v1 scope. |
| `--exit-on` predicate grammar | Kept minimal; finalised at implementation. |
