# ADR: wargs Design Decisions

**Date:** 2026-03-31
**Status:** Proposed
**Related:** [wargs design](2026-03-31-wargs-design.md)

---

## 1. Line-delimited input by default

**Context:** POSIX xargs defaults to whitespace splitting with quote handling, which breaks on filenames with spaces — the most common xargs footgun.

**Decision:** Default to line-delimited input (one item per `\n`).

**Rationale:** Line-per-item is what users actually mean 95% of the time. Modern tools (`find -print0`, `git ls-files`) produce line or null-delimited output. Whitespace splitting is only needed for strict POSIX compat.

**Trade-offs Accepted:** Not drop-in compatible with scripts that rely on xargs whitespace splitting. Mitigated by `--compat` flag.

**Options Considered:**
- Whitespace default (match POSIX): rejected — reproduces the most common xargs bug
- Auto-detect: rejected — unreliable heuristics, surprising behaviour

---

## 2. Smart `{}` placeholder detection

**Context:** GNU xargs uses `-I {}` to enable placeholder substitution; without it, args are appended. Two-mode behaviour with an explicit flag.

**Decision:** Automatically detect: if `{}` appears anywhere in the command template, use substitution mode; otherwise, append mode. No flag needed.

**Rationale:** Does the right thing based on what the user wrote. No flag to remember. `wargs echo` appends, `wargs echo "processing {}"` substitutes.

**Trade-offs Accepted:** Can't use literal `{}` in a command without triggering substitution. Vanishingly rare in practice. If needed later, a `-I` flag can override the placeholder string.

**Options Considered:**
- Always require `{}` (no append mode): rejected — breaks xargs muscle memory
- Require `-I` flag: rejected — unnecessary ceremony for the common case

## 3. Pipeline architecture (InputReader → CommandBuilder → JobRunner)

**Context:** Need to read stdin, build commands, execute them with parallelism, and collect results.

**Decision:** Three composable stages, each independently testable.

**Rationale:** Matches existing Winix library patterns (logic in class library, I/O in console app). Each stage has clear inputs/outputs. InputReader tested with MemoryStream, CommandBuilder is pure functions, JobRunner tested with mock commands.

**Trade-offs Accepted:** More files than a single-class approach. Worth it for testability.

**Options Considered:**
- Single orchestrator class: rejected — hard to test stages in isolation
- System.Threading.Channels reactive pipeline: rejected — over-engineered, no backpressure problem to solve

---

## 4. Job-buffered output as default

**Context:** Parallel execution means concurrent output. Need a strategy for interleaving.

**Decision:** Default to job-buffered (capture per-job, print atomically on completion). Offer `--line-buffered` and `--keep-order` flags.

**Rationale:** Job-buffered gives clean, non-interleaved output for the common case. Line-buffered is there for fast feedback when you need it. Keep-order for deterministic scripting.

**Trade-offs Accepted:** No output visible until a job completes (could be slow for long-running commands). `--line-buffered` is the escape hatch.

**Options Considered:**
- Line-buffered default (match xargs): rejected — messy with parallel jobs
- Keep-order default: rejected — first slow job blocks all output

---

## 5. Exit code 123 for child failures

**Context:** Need an exit code convention for "some children failed."

**Decision:** Use 123, matching GNU xargs convention.

**Rationale:** Users migrating from xargs expect this. 124 for fail-fast abort distinguishes "I stopped early" from "everything ran, some failed."

**Trade-offs Accepted:** Locks us into the xargs exit code space. Acceptable — these are well-understood conventions.

**Options Considered:**
- Exit with the highest child exit code: rejected — loses the "wargs itself succeeded" vs "a child failed" distinction
- Exit 1 for any failure: rejected — too generic, conflicts with tools that use 1 for specific meanings

---

## 6. Confirm reads from /dev/tty (CON on Windows), not stdin

**Context:** `-p` confirm mode needs to prompt the user, but stdin is the input pipe.

**Decision:** Read confirmation from `/dev/tty` (Unix) or `CON` (Windows).

**Rationale:** This is how GNU xargs does it, and it's the only correct approach when stdin is a pipe. Opening the controlling terminal directly bypasses the pipe.

**Trade-offs Accepted:** Won't work in environments without a controlling terminal (e.g., headless CI). That's fine — confirm mode is for interactive use.

**Options Considered:**
- Read from stdin after input is consumed: rejected — can't know when input ends in a streaming context
- Require a separate flag to specify the prompt source: rejected — over-engineering

---

## 7. Name: wargs

**Context:** Needed a name for the xargs replacement. Considered `each`, `foreach`, `wargs`.

**Decision:** `wargs` (Winix args).

**Rationale:** `each` and `foreach` are common shell builtins/keywords that could conflict. `wargs` is clearly a Winix tool, unlikely to collide, and the xargs heritage is obvious.

**Trade-offs Accepted:** Less "natural language" than `each`. The xargs echo is a feature, not a bug — users know what this tool does from the name.

---

## Decisions Explicitly Deferred

| Topic | Why deferred |
|-------|-------------|
| Custom placeholder string (`-I`) | `{}` is sufficient for v1; can add without breaking changes |
| Multi-placeholder CSV splitting (`{1}`, `{2}`) | Niche use case, adds parser complexity |
| Retry on failure | Can layer on top without changing core architecture |
| Rate limiting / throttle | No current use case identified |
| ARG_MAX auto-chunking | Unix artefact, not relevant on Windows |
